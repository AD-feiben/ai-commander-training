## 一、AI + Git 工作流

### 1.1 传统工作流 vs AI 增强工作流

```
传统 Git 工作流：
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 编写代码  │───→│ git diff │───→│ 手动编写 │───→│ git commit│
│          │    │ 查看变更 │    │ commit   │    │          │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
     │                                               │
     └───────────────────┬───────────────────────────┘
                         ▼
                  ┌──────────┐
                  │  push 到 │
                  │  remote  │
                  └──────────┘

AI 增强工作流：
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 编写代码  │───→│ git diff │───→│ AI 生成  │───→│ git commit│
│          │    │ 查看变更 │    │ commit   │    │ (自动)   │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
     │                                               │
     └───────────────────┬───────────────────────────┘
                         ▼
                  ┌──────────┐
                  │ AI 代码  │
                  │ 审查     │
                  └────┬─────┘
                       ▼
                  ┌──────────┐
                  │  push 到 │
                  │  remote  │
                  └──────────┘
```

### 1.2 AI 在 Git 工作流中的应用

| 环节 | 传统方式 | AI 增强方式 |
|------|----------|-------------|
| **提交信息** | 手动编写 | AI 根据 diff 自动生成 |
| **代码审查** | 人工审查 | AI 预审 + 人工复核 |
| **分支管理** | 手动创建/合并 | AI 建议分支策略 |
| **冲突解决** | 手动解决 | AI 辅助冲突解决 |
| **提交建议** | 无 | AI 建议何时提交 |

## 二、智能提交信息生成

### 2.1 为什么提交信息很重要

```
好的提交信息：
feat(auth): 添加用户登录功能

- 实现 JWT Token 认证
- 添加登录表单验证
- 集成密码加密

Closes #123

坏的提交信息：
更新代码
fix bug
修改
```

### 2.2 AI 生成提交信息

**提示词模板：**

```markdown
请为以下代码变更生成符合 Conventional Commits 规范的提交信息。

变更内容：
```diff
[git diff 内容]
```

要求：
1. 使用 Conventional Commits 格式
2. 标题不超过 50 个字符
3. 正文说明变更原因和内容
4. 如有破坏性变更，标注 BREAKING CHANGE

输出格式：
type(scope): subject

body

footer
```

**示例输出：**

```
feat(api): 添加用户注册接口

- 实现 /api/register 端点
- 添加邮箱格式验证
- 集成 bcrypt 密码加密
- 添加重复用户检测

Closes #45
```

### 2.3 实现代码

```typescript
class CommitMessageGenerator {
  private openai: OpenAI;
  
  constructor(apiKey: string) {
    this.openai = new OpenAI({ apiKey });
  }
  
  async generate(diff: string): Promise<string> {
    const prompt = `
请为以下代码变更生成符合 Conventional Commits 规范的提交信息。

变更内容：
\`\`\`diff
${diff}
\`\`\`

要求：
1. type 可选值: feat, fix, docs, style, refactor, test, chore
2. scope 应该是受影响的模块名
3. subject 使用祈使句，首字母小写，不加句号
4. body 详细说明变更内容和原因
5. footer 用于引用 issue 或标注破坏性变更

只输出提交信息，不要其他内容。
    `;
    
    const response = await this.openai.chat.completions.create({
      model: 'gpt-4',
      messages: [
        {
          role: 'system',
          content: '你是 Git 专家，擅长编写规范的提交信息。'
        },
        {
          role: 'user',
          content: prompt
        }
      ],
      temperature: 0.3
    });
    
    return response.choices[0].message.content.trim();
  }
}

// 使用示例
const generator = new CommitMessageGenerator(process.env.OPENAI_API_KEY);
const diff = await execAsync('git diff --staged');
const message = await generator.generate(diff);
await execAsync(`git commit -m "${message}"`);
```

## 三、自动化分支管理

### 3.1 分支命名规范

```
AI 建议的分支命名：

功能分支：
- 输入：实现用户登录
- 输出：feat/user-login

修复分支：
- 输入：修复登录页面的样式问题
- 输出：fix/login-page-style

重构分支：
- 输入：重构用户认证模块
- 输出：refactor/auth-module
```

### 3.2 智能分支管理

```typescript
class BranchManager {
  async suggestBranchName(task: string): Promise<string> {
    const prompt = `
根据以下任务描述，建议一个符合 Git Flow 规范的分支名称。

任务：${task}

要求：
1. 使用 kebab-case（短横线连接）
2. 前缀选择：feat/, fix/, docs/, refactor/, test/
3. 名称简洁明了，不超过 50 字符

只输出分支名称，不要其他内容。
    `;
    
    // 调用 AI 获取建议
    const branchName = await this.aiComplete(prompt);
    return branchName.trim();
  }
  
  async suggestMergeStrategy(
    sourceBranch: string,
    targetBranch: string
  ): Promise<MergeStrategy> {
    const prompt = `
分析以下分支合并场景，建议最佳的合并策略。

源分支：${sourceBranch}
目标分支：${targetBranch}

请考虑：
1. 分支类型（feature/fix/release）
2. 是否需要保留提交历史
3. 是否需要压缩提交

输出格式：
strategy: merge | squash | rebase
reason: 选择原因
    `;
    
    const response = await this.aiComplete(prompt);
    return this.parseStrategy(response);
  }
}
```

## 四、AI 代码审查集成

### 4.1 审查流程

```
代码提交前审查流程：

┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 代码变更  │───→│ AI 预审  │───→│ 问题修复  │───→│ 人工审查 │
│          │    │          │    │          │    │          │
└──────────┘    └────┬─────┘    └──────────┘    └────┬─────┘
                     │                               │
                     ↓                               ↓
               ┌──────────┐                   ┌──────────┐
               │ 发现问题 │                   │ 通过/拒绝 │
               └──────────┘                   └──────────┘
```

### 4.2 Git Hook 集成

```bash
#!/bin/sh
# .git/hooks/pre-commit

# 运行 AI 代码审查
echo "🤖 正在运行 AI 代码审查..."

# 获取暂存区的文件
STAGED_FILES=$(git diff --cached --name-only --diff-filter=ACM | grep -E '\.(js|ts|jsx|tsx)$')

if [ -n "$STAGED_FILES" ]; then
  # 调用 AI 审查工具
  npx ai-code-review --files "$STAGED_FILES" --format json > .ai-review-result.json
  
  # 检查是否有严重问题
  CRITICAL_ISSUES=$(cat .ai-review-result.json | jq '.criticalIssues')
  
  if [ "$CRITICAL_ISSUES" -gt 0 ]; then
    echo "❌ 发现 ${CRITICAL_ISSUES} 个严重问题，请修复后重新提交"
    cat .ai-review-result.json | jq '.issues[] | select(.severity == "critical")'
    exit 1
  fi
  
  # 显示警告
  WARNINGS=$(cat .ai-review-result.json | jq '.warnings')
  if [ "$WARNINGS" -gt 0 ]; then
    echo "⚠️  发现 ${WARNINGS} 个警告"
  fi
fi

echo "✅ AI 审查通过"
exit 0
```

### 4.3 PR 审查自动化

```yaml
# .github/workflows/ai-code-review.yml
name: AI Code Review

on:
  pull_request:
    types: [opened, synchronize]

jobs:
  ai-review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
        with:
          fetch-depth: 0

      - name: Setup Node.js
        uses: actions/setup-node@v3
        with:
          node-version: '18'

      - name: Get PR diff
        id: diff
        run: |
          git diff origin/${{ github.base_ref }}...HEAD > pr.diff

      - name: AI Code Review
        id: review
        env:
          OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
        run: |
          npx ai-code-review \
            --diff pr.diff \
            --output review.md

      - name: Post Review Comment
        uses: actions/github-script@v6
        with:
          script: |
            const fs = require('fs');
            const review = fs.readFileSync('review.md', 'utf8');
            
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: `## 🤖 AI Code Review\n\n${review}`
            });
```

## 五、本节小结

### 核心概念

1. **AI 可以大幅提升 Git 操作效率**，从提交信息生成到代码审查

2. **智能提交信息** 遵循 Conventional Commits 规范

3. **自动化分支管理** 根据任务类型推荐分支策略

4. **AI 代码审查** 可以在提交前和 PR 时自动运行

### 最佳实践

- 使用 AI 生成提交信息，但保持人工审核
- 将 AI 审查作为预审，人工审查作为终审
- 建立团队规范，统一 AI 工具的使用方式
