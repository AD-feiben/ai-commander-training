# L6-2: 任务分解与分配

> 化繁为简，让复杂任务变得可管理

## 本节导读

面对一个复杂的需求，如何将其拆解为可执行的子任务？如何合理分配给不同的 Agent？这是 Multi-Agent 系统的核心挑战。本节课将深入讲解任务分解的艺术和任务分配的科学。

通过本节课，你将学会：
- 任务分解的原则和方法
- 任务分配的算法和策略
- 依赖管理和调度优化
- 实战：实现一个任务调度器

## 一、任务分解的艺术

### 1.1 为什么需要任务分解

**复杂任务的困境：**

```
用户请求：开发一个电商平台
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│  不分解直接执行：                                             │
│  - Agent 上下文超载                                           │
│  - 遗漏关键功能                                               │
│  - 质量难以保证                                               │
│  - 无法并行处理                                               │
└─────────────────────────────────────────────────────────────┘

分解后：
    │
    ├──→ 用户模块
    ├──→ 商品模块
    ├──→ 订单模块
    ├──→ 支付模块
    └──→ 物流模块
         │
         ├──→ 需求分析
         ├──→ 架构设计
         ├──→ 接口定义
         ├──→ 代码实现
         └──→ 测试验证
```

### 1.2 任务分解的原则

**MECE 原则（Mutually Exclusive, Collectively Exhaustive）：**

```
任务分解
    │
    ├──→ 子任务 A（独立、不重叠）
    ├──→ 子任务 B（独立、不重叠）
    └──→ 子任务 C（独立、不重叠）
    
要求：
- 相互独立：子任务之间没有重叠
- 完全穷尽：所有子任务覆盖完整需求
```

**分解粒度原则：**

| 粒度 | 说明 | 示例 |
|------|------|------|
| **太粗** | 难以执行，需要进一步分解 | "开发电商平台" |
| **适中** | 可独立完成，有明确输出 | "实现用户注册 API" |
| **太细** | 管理成本高，效率低 | "定义 username 变量" |

**推荐的分解层级：**

```
项目（Project）
    │
    ├──→ 模块（Module）- 如：用户模块、订单模块
    │       │
    │       ├──→ 功能（Feature）- 如：用户注册、用户登录
    │       │       │
    │       │       ├──→ 任务（Task）- 如：API 设计、接口实现
    │       │       │       │
    │       │       │       └──→ 子任务（Subtask）- 如：参数校验、数据库操作
```

## 二、任务依赖管理

### 2.1 依赖类型

```
依赖类型：

1. 完成-开始（FS）：A 完成后 B 才能开始
   A ──→ B
   
2. 开始-开始（SS）：A 开始后 B 才能开始
   A ──→ B
   ↓
   
3. 完成-完成（FF）：A 完成后 B 才能完成
   A ──→ B
        ↓
        
4. 开始-完成（SF）：A 开始后 B 才能完成
   A ──→ B
   ↓     ↓
```

### 2.2 依赖图（DAG）

**示例：项目启动依赖图**

```
┌─────────────────────────────────────────────────────────────┐
│                    任务依赖图（DAG）                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  [需求分析] ──→ [架构设计] ──→ [数据库设计]                 │
│       │              │               │                      │
│       │              ↓               ↓                      │
│       │         [API 设计] ←── [接口定义]                   │
│       │              │               │                      │
│       ↓              ↓               ↓                      │
│  [UI 设计] ──→ [前端开发] ←── [后端开发]                    │
│                      │               │                      │
│                      └───────┬───────┘                      │
│                              ↓                              │
│                         [接口联调]                          │
│                              │                              │
│                              ↓                              │
│                         [集成测试]                          │
│                              │                              │
│                              ↓                              │
│                         [部署上线]                          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.3 关键路径分析

```
关键路径：决定项目最短完成时间的路径

示例：
路径 1: 需求分析 → UI 设计 → 前端开发 → 接口联调 → 集成测试 → 部署
       (2d)      (3d)       (5d)        (2d)        (3d)       (1d) = 16d

路径 2: 需求分析 → 架构设计 → API 设计 → 后端开发 → 接口联调 → 集成测试 → 部署
       (2d)      (3d)        (2d)       (5d)        (2d)        (3d)       (1d) = 18d

路径 3: 需求分析 → 架构设计 → 数据库设计 → 接口定义 → 后端开发 → ...
       (2d)      (3d)         (2d)        (1d)        (5d)      ... = 19d

关键路径：路径 3（19天）
优化方向：缩短关键路径上的任务时间
```

## 三、任务分配策略

### 3.1 分配考虑因素

```
任务分配决策树：

任务特性
    │
    ├──→ 复杂度
    │       ├──→ 高 → 分配给资深 Agent
    │       └──→ 低 → 分配给初级 Agent
    │
    ├──→ 紧急度
    │       ├──→ 高 → 分配给空闲 Agent
    │       └──→ 低 → 按正常流程分配
    │
    ├──→ 专业性
    │       ├──→ 前端 → 分配给前端 Agent
    │       ├──→ 后端 → 分配给后端 Agent
    │       └──→ 通用 → 分配给任意 Agent
    │
    └──→ 依赖性
            ├──→ 依赖其他任务 → 等待依赖完成
            └──→ 独立任务 → 立即分配
```

### 3.2 分配算法

#### 算法 1：轮询（Round Robin）

```typescript
class RoundRobinScheduler {
  private agents: Agent[];
  private currentIndex = 0;
  
  assignTask(task: Task): Agent {
    const agent = this.agents[this.currentIndex];
    this.currentIndex = (this.currentIndex + 1) % this.agents.length;
    return agent;
  }
}

// 优点：简单公平
// 缺点：不考虑 Agent 能力和负载
```

#### 算法 2：能力匹配

```typescript
class CapabilityBasedScheduler {
  assignTask(task: Task): Agent {
    // 根据任务需求和 Agent 能力匹配
    const candidates = this.agents.filter(agent => 
      agent.skills.includes(task.requiredSkill)
    );
    
    // 选择负载最低的
    return candidates.sort((a, b) => 
      a.currentLoad - b.currentLoad
    )[0];
  }
}

// 优点：专业对口
// 缺点：可能出现某些 Agent 过载
```

#### 算法 3：负载均衡

```typescript
class LoadBalancerScheduler {
  assignTask(task: Task): Agent {
    // 综合考虑能力和负载
    const scoredAgents = this.agents.map(agent => ({
      agent,
      score: this.calculateScore(agent, task)
    }));
    
    return scoredAgents.sort((a, b) => b.score - a.score)[0].agent;
  }
  
  private calculateScore(agent: Agent, task: Task): number {
    const capabilityScore = agent.skills.includes(task.requiredSkill) ? 100 : 0;
    const loadScore = 100 - agent.currentLoad;
    const performanceScore = agent.averagePerformance;
    
    return capabilityScore * 0.5 + loadScore * 0.3 + performanceScore * 0.2;
  }
}

// 优点：综合考虑多因素
// 缺点：计算复杂度较高
```

### 3.3 动态调整

```
初始分配
    │
    ▼
监控执行 ──→ 发现问题 ──→ 重新分配
    │                        ↑
    └──→ 正常 ──→ 完成 ──────┘

调整触发条件：
1. Agent 执行失败
2. Agent 执行超时
3. 发现更优分配方案
4. 新增 Agent 加入
```

## 四、实战：任务调度器实现

### 4.1 数据模型

```typescript
// 任务定义
interface Task {
  id: string;
  name: string;
  description: string;
  type: 'analysis' | 'design' | 'development' | 'review' | 'test';
  priority: 'low' | 'medium' | 'high' | 'critical';
  estimatedDuration: number; // 分钟
  requiredSkills: string[];
  dependencies: string[]; // 依赖的任务 ID
  input?: any;
  output?: any;
  status: 'pending' | 'assigned' | 'running' | 'completed' | 'failed';
  assignedTo?: string; // Agent ID
}

// Agent 定义
interface Agent {
  id: string;
  name: string;
  role: string;
  skills: string[];
  maxConcurrentTasks: number;
  currentTasks: string[];
  performance: {
    successRate: number;
    averageCompletionTime: number;
  };
}

// 调度计划
interface Schedule {
  taskId: string;
  agentId: string;
  startTime: Date;
  endTime: Date;
  status: 'scheduled' | 'running' | 'completed';
}
```

### 4.2 调度器实现

```typescript
class TaskScheduler {
  private tasks: Map<string, Task> = new Map();
  private agents: Map<string, Agent> = new Map();
  private schedules: Schedule[] = [];
  
  // 添加任务
  addTask(task: Task): void {
    this.tasks.set(task.id, task);
  }
  
  // 添加 Agent
  addAgent(agent: Agent): void {
    this.agents.set(agent.id, agent);
  }
  
  // 构建依赖图
  buildDependencyGraph(): Map<string, string[]> {
    const graph = new Map<string, string[]>();
    
    for (const [id, task] of this.tasks) {
      graph.set(id, task.dependencies);
    }
    
    return graph;
  }
  
  // 拓扑排序（确定执行顺序）
  topologicalSort(): string[] {
    const graph = this.buildDependencyGraph();
    const visited = new Set<string>();
    const temp = new Set<string>();
    const result: string[] = [];
    
    const visit = (node: string) => {
      if (temp.has(node)) {
        throw new Error(`检测到循环依赖: ${node}`);
      }
      
      if (visited.has(node)) return;
      
      temp.add(node);
      
      const dependencies = graph.get(node) || [];
      for (const dep of dependencies) {
        visit(dep);
      }
      
      temp.delete(node);
      visited.add(node);
      result.push(node);
    };
    
    for (const node of graph.keys()) {
      visit(node);
    }
    
    return result;
  }
  
  // 选择最佳 Agent
  selectBestAgent(task: Task): Agent | null {
    const availableAgents = Array.from(this.agents.values())
      .filter(agent => 
        // 检查技能匹配
        task.requiredSkills.every(skill => 
          agent.skills.includes(skill)
        ) &&
        // 检查并发限制
        agent.currentTasks.length < agent.maxConcurrentTasks
      );
    
    if (availableAgents.length === 0) return null;
    
    // 按负载和性能评分
    return availableAgents.sort((a, b) => {
      const scoreA = this.calculateAgentScore(a, task);
      const scoreB = this.calculateAgentScore(b, task);
      return scoreB - scoreA;
    })[0];
  }
  
  private calculateAgentScore(agent: Agent, task: Task): number {
    const loadFactor = 1 - (agent.currentTasks.length / agent.maxConcurrentTasks);
    const performanceFactor = agent.performance.successRate;
    const speedFactor = 1 / (agent.performance.averageCompletionTime + 1);
    
    return loadFactor * 0.4 + performanceFactor * 0.4 + speedFactor * 0.2;
  }
  
  // 执行调度
  async execute(): Promise<void> {
    const executionOrder = this.topologicalSort();
    
    for (const taskId of executionOrder) {
      const task = this.tasks.get(taskId)!;
      
      // 等待依赖完成
      await this.waitForDependencies(task);
      
      // 分配 Agent
      const agent = this.selectBestAgent(task);
      if (!agent) {
        throw new Error(`没有可用的 Agent 处理任务: ${task.name}`);
      }
      
      // 执行任务
      await this.executeTask(task, agent);
    }
  }
  
  private async waitForDependencies(task: Task): Promise<void> {
    const checkDependencies = () => {
      return task.dependencies.every(depId => {
        const depTask = this.tasks.get(depId);
        return depTask?.status === 'completed';
      });
    };
    
    while (!checkDependencies()) {
      await new Promise(resolve => setTimeout(resolve, 1000));
    }
  }
  
  private async executeTask(task: Task, agent: Agent): Promise<void> {
    task.status = 'running';
    task.assignedTo = agent.id;
    agent.currentTasks.push(task.id);
    
    try {
      // 实际执行（调用 Agent）
      await this.callAgent(agent, task);
      
      task.status = 'completed';
    } catch (error) {
      task.status = 'failed';
      throw error;
    } finally {
      // 释放 Agent
      agent.currentTasks = agent.currentTasks.filter(id => id !== task.id);
    }
  }
  
  private async callAgent(agent: Agent, task: Task): Promise<void> {
    // 实际调用 Agent 的接口
    console.log(`Agent ${agent.name} 正在执行任务: ${task.name}`);
    // ... 实际调用逻辑
  }
  
  // 获取调度状态
  getStatus(): {
    pending: number;
    running: number;
    completed: number;
    failed: number;
  } {
    const tasks = Array.from(this.tasks.values());
    return {
      pending: tasks.filter(t => t.status === 'pending').length,
      running: tasks.filter(t => t.status === 'running').length,
      completed: tasks.filter(t => t.status === 'completed').length,
      failed: tasks.filter(t => t.status === 'failed').length
    };
  }
}
```

## 五、调度优化策略

### 5.1 并行化优化

```
串行执行：
A ──→ B ──→ C ──→ D
2h    3h    2h    1h = 8h

并行优化：
A ──→ B ──┬──→ C ──→ D
          │
          └──→ E ──→ F
2h    3h   2h    1h = 8h (无优化)

优化后：
A ──→ B ──→ C ──→ D
2h    3h    2h    1h = 8h
└──→ E ──→ F
     2h    1h = 3h (并行)

总时间：max(8h, 2h+3h) = 8h → 5h (A+E 并行)
```

### 5.2 优先级调度

```typescript
class PriorityScheduler extends TaskScheduler {
  // 优先级权重
  private priorityWeights = {
    critical: 1000,
    high: 100,
    medium: 10,
    low: 1
  };
  
  selectBestAgent(task: Task): Agent | null {
    // 先按优先级排序
    const priorityScore = this.priorityWeights[task.priority];
    
    // 在优先级高的任务中分配
    return super.selectBestAgent(task);
  }
}
```

### 5.3 失败重试

```typescript
class ResilientScheduler extends TaskScheduler {
  private maxRetries = 3;
  private retryDelay = 5000; // 5秒
  
  private async executeTask(task: Task, agent: Agent): Promise<void> {
    let retries = 0;
    
    while (retries < this.maxRetries) {
      try {
        await super.executeTask(task, agent);
        return;
      } catch (error) {
        retries++;
        
        if (retries >= this.maxRetries) {
          task.status = 'failed';
          throw error;
        }
        
        console.log(`任务 ${task.name} 失败，${retries} 秒后重试...`);
        await new Promise(resolve => setTimeout(resolve, this.retryDelay));
        
        // 尝试更换 Agent
        const newAgent = this.selectBestAgent(task);
        if (newAgent && newAgent.id !== agent.id) {
          agent = newAgent;
        }
      }
    }
  }
}
```

## 六、本节小结

### 核心概念

1. **任务分解** 是管理复杂任务的基础，遵循 MECE 原则

2. **依赖管理** 通过 DAG 识别任务执行顺序和关键路径

3. **任务分配** 需要综合考虑能力匹配、负载均衡和优先级

4. **调度优化** 包括并行化、优先级调度和失败重试

### 关键要点

- 任务分解要适度，太粗难以执行，太细管理成本高
- 依赖图帮助识别并行机会和关键路径
- 动态调度比静态分配更能适应变化
- 失败处理机制保证系统健壮性
