# L6-2: Task Decomposition and Assignment

> Breaking complexity into manageability

## Section Overview

When facing a complex requirement, how do you split it into executable subtasks? How do you assign them to different Agents reasonably? This is the core challenge of Multi-Agent systems. This lesson will dive deep into the art of task decomposition and the science of task assignment.

Through this lesson, you will learn:
- Principles and methods of task decomposition
- Algorithms and strategies for task assignment
- Dependency management and scheduling optimization
- Practice: Implement a task scheduler

## 1. The Art of Task Decomposition

### 1.1 Why Task Decomposition is Needed

**The Dilemma of Complex Tasks:**

```
User Request: Develop an e-commerce platform
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│  Execute without decomposition:                              │
│  - Agent context overload                                    │
│  - Missing key features                                      │
│  - Quality hard to guarantee                                  │
│  - Cannot process in parallel                                 │
└─────────────────────────────────────────────────────────────┘

After decomposition:
    │
    ├──→ User Module
    ├──→ Product Module
    ├──→ Order Module
    ├──→ Payment Module
    └──→ Logistics Module
         │
         ├──→ Requirements Analysis
         ├──→ Architecture Design
         ├──→ Interface Definition
         ├──→ Code Implementation
         └──→ Testing and Validation
```

### 1.2 Principles of Task Decomposition

**MECE Principle (Mutually Exclusive, Collectively Exhaustive):**

```
Task Decomposition
    │
    ├──→ Subtask A (Independent, Non-overlapping)
    ├──→ Subtask B (Independent, Non-overlapping)
    └──→ Subtask C (Independent, Non-overlapping)

Requirements:
- Mutually Exclusive: No overlap between subtasks
- Collectively Exhaustive: All subtasks cover complete requirements
```

**Decomposition Granularity Principle:**

| Granularity | Description | Example |
|-------------|-------------|---------|
| **Too Coarse** | Hard to execute, needs further decomposition | "Develop e-commerce platform" |
| **Appropriate** | Can be completed independently with clear output | "Implement user registration API" |
| **Too Fine** | High management cost, low efficiency | "Define username variable" |

**Recommended Decomposition Levels:**

```
Project
    │
    ├──→ Module - e.g., User Module, Order Module
    │       │
    │       ├──→ Feature - e.g., User Registration, User Login
    │       │       │
    │       │       ├──→ Task - e.g., API Design, Interface Implementation
    │       │       │       │
    │       │       │       └──→ Subtask - e.g., Parameter Validation, Database Operation
```

## 2. Task Dependency Management

### 2.1 Dependency Types

```
Dependency Types:

1. Finish-to-Start (FS): B can only start after A finishes
   A ──→ B

2. Start-to-Start (SS): B can only start after A starts
   A ──→ B
   ↓

3. Finish-to-Finish (FF): B can only finish after A finishes
   A ──→ B
        ↓

4. Start-to-Finish (SF): B can only finish after A starts
   A ──→ B
   ↓     ↓
```

### 2.2 Dependency Graph (DAG)

**Example: Project Startup Dependency Graph**

```
┌─────────────────────────────────────────────────────────────┐
│                    Task Dependency Graph (DAG)              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  [Requirements] ──→ [Architecture] ──→ [Database Design]  │
│       │              │               │                      │
│       │              ↓               ↓                      │
│       │         [API Design] ←── [Interface Definition]     │
│       │              │               │                      │
│       ↓              ↓               ↓                      │
│  [UI Design] ──→ [Frontend Dev] ←── [Backend Dev]         │
│                      │               │                      │
│                      └───────┬───────┘                      │
│                              ↓                              │
│                         [Interface Integration]            │
│                              │                              │
│                              ↓                              │
│                         [Integration Testing]              │
│                              │                              │
│                              ↓                              │
│                         [Deployment]                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.3 Critical Path Analysis

```
Critical Path: The path that determines the shortest project completion time

Example:
Path 1: Requirements → UI Design → Frontend Dev → Interface Integration → Integration Testing → Deployment
       (2d)      (3d)       (5d)        (2d)        (3d)       (1d) = 16d

Path 2: Requirements → Architecture → API Design → Backend Dev → Interface Integration → Integration Testing → Deployment
       (2d)      (3d)        (2d)       (5d)        (2d)        (3d)       (1d) = 18d

Path 3: Requirements → Architecture → Database Design → Interface Definition → Backend Dev → ...
       (2d)      (3d)         (2d)        (1d)        (5d)      ... = 19d

Critical Path: Path 3 (19 days)
Optimization Direction: Shorten task time on critical path
```

## 3. Task Assignment Strategies

### 3.1 Assignment Considerations

```
Task Assignment Decision Tree:

Task Characteristics
    │
    ├──→ Complexity
    │       ├──→ High → Assign to senior Agent
    │       └──→ Low → Assign to junior Agent
    │
    ├──→ Urgency
    │       ├──→ High → Assign to idle Agent
    │       └──→ Low → Assign via normal process
    │
    ├──→ Specialization
    │       ├──→ Frontend → Assign to frontend Agent
    │       ├──→ Backend → Assign to backend Agent
    │       └──→ General → Assign to any Agent
    │
    └──→ Dependencies
            ├──→ Depends on other tasks → Wait for dependencies
            └──→ Independent task → Assign immediately
```

### 3.2 Assignment Algorithms

#### Algorithm 1: Round Robin

```typescript
class RoundRobinScheduler {
  private agents: Agent[];
  private currentIndex = 0;

  assignTask(task: Task): Agent {
    const agent = this.agents[this.currentIndex];
    this.currentIndex = (this.currentIndex + 1) % this.agents.length;
    return agent;
  }
}

// Pros: Simple and fair
// Cons: Doesn't consider Agent capability or load
```

#### Algorithm 2: Capability Matching

```typescript
class CapabilityBasedScheduler {
  assignTask(task: Task): Agent {
    // Match based on task requirements and Agent capabilities
    const candidates = this.agents.filter(agent =>
      agent.skills.includes(task.requiredSkill)
    );

    // Select the one with lowest load
    return candidates.sort((a, b) =>
      a.currentLoad - b.currentLoad
    )[0];
  }
}

// Pros: Professional matching
// Cons: Some Agents may become overloaded
```

#### Algorithm 3: Load Balancing

```typescript
class LoadBalancerScheduler {
  assignTask(task: Task): Agent {
    // Consider both capability and load
    const scoredAgents = this.agents.map(agent => ({
      agent,
      score: this.calculateScore(agent, task)
    }));

    return scoredAgents.sort((a, b) => b.score - a.score)[0].agent;
  }

  private calculateScore(agent: Agent, task: Task): number {
    const capabilityScore = agent.skills.includes(task.requiredSkill) ? 100 : 0;
    const loadScore = 100 - agent.currentLoad;
    const performanceScore = agent.averagePerformance;

    return capabilityScore * 0.5 + loadScore * 0.3 + performanceScore * 0.2;
  }
}

// Pros: Considers multiple factors
// Cons: Higher computational complexity
```

### 3.3 Dynamic Adjustment

```
Initial Assignment
    │
    ▼
Monitor Execution ──→发现问题 ──→ Reassign
    │                        ↑
    └──→ Normal ──→ Complete ──┘

Adjustment Triggers:
1. Agent execution failure
2. Agent execution timeout
3. Better assignment discovered
4. New Agent joins
```

### 4.1 Data Model

```typescript
// Task Definition
interface Task {
  id: string;

  type: 'analysis' | 'design' | 'development' | 'review' | 'test';
  priority: 'low' | 'medium' | 'high' | 'critical';

}

// Agent Definition
interface Agent {
  id: string;
  name: string;
  role: string;
  skills: string[];
  maxConcurrentTasks: number;
  currentTasks: string[];
  performance: {
    successRate: number;
    averageCompletionTime: number;
  };
}

// Schedule
interface Schedule {
  taskId: string;
  agentId: string;
  startTime: Date;
  endTime: Date;
  status: 'scheduled' | 'running' | 'completed';
}
```

### 4.2 Scheduler Implementation

```typescript
class TaskScheduler {
  private tasks: Map<string, Task> = new Map();
  private agents: Map<string, Agent> = new Map();
  private schedules: Schedule[] = [];

  // Add task
  addTask(task: Task): void {
    this.tasks.set(task.id, task);
  }

  // Add Agent
  addAgent(agent: Agent): void {
    this.agents.set(agent.id, agent);
  }

  // Build dependency graph
  buildDependencyGraph(): Map<string, string[]> {
    const graph = new Map<string, string[]>();

    for (const [id, task] of this.tasks) {
      graph.set(id, task.dependencies);
    }

    return graph;
  }

  // Topological sort (determine execution order)
  topologicalSort(): string[] {
    const graph = this.buildDependencyGraph();
    const visited = new Set<string>();
    const temp = new Set<string>();
    const result: string[] = [];

    const visit = (node: string) => {
      if (temp.has(node)) {
        throw new Error(`Circular dependency detected: ${node}`);
      }

      if (visited.has(node)) return;

      temp.add(node);

      const dependencies = graph.get(node) || [];
      for (const dep of dependencies) {
        visit(dep);
      }

      temp.delete(node);
      visited.add(node);
      result.push(node);
    };

    for (const node of graph.keys()) {
      visit(node);
    }

    return result;
  }

  // Select best Agent
  selectBestAgent(task: Task): Agent | null {
    const availableAgents = Array.from(this.agents.values())
      .filter(agent =>
        // Check skill matching
        task.requiredSkills.every(skill =>
          agent.skills.includes(skill)
        ) &&
        // Check concurrent limit
        agent.currentTasks.length < agent.maxConcurrentTasks
      );

    if (availableAgents.length === 0) return null;

    // Score by load and performance
    return availableAgents.sort((a, b) => {
      const scoreA = this.calculateAgentScore(a, task);
      const scoreB = this.calculateAgentScore(b, task);
      return scoreB - scoreA;
    })[0];
  }

  private calculateAgentScore(agent: Agent, task: Task): number {
    const loadFactor = 1 - (agent.currentTasks.length / agent.maxConcurrentTasks);
    const performanceFactor = agent.performance.successRate;
    const speedFactor = 1 / (agent.performance.averageCompletionTime + 1);

    return loadFactor * 0.4 + performanceFactor * 0.4 + speedFactor * 0.2;
  }

  // Execute schedule
  async execute(): Promise<void> {
    const executionOrder = this.topologicalSort();

    for (const taskId of executionOrder) {
      const task = this.tasks.get(taskId)!;

      // Wait for dependencies
      await this.waitForDependencies(task);

      // Assign Agent
      const agent = this.selectBestAgent(task);
      if (!agent) {
        throw new Error(`No available Agent for task: ${task.name}`);
      }

      // Execute task
      await this.executeTask(task, agent);
    }
  }

  private async waitForDependencies(task: Task): Promise<void> {
    const checkDependencies = () => {
      return task.dependencies.every(depId => {
        const depTask = this.tasks.get(depId);
        return depTask?.status === 'completed';
      });
    };

    while (!checkDependencies()) {
      await new Promise(resolve => setTimeout(resolve, 1000));
    }
  }

  private async executeTask(task: Task, agent: Agent): Promise<void> {
    task.status = 'running';
    task.assignedTo = agent.id;
    agent.currentTasks.push(task.id);

    try {
      // Actual execution (call Agent)
      await this.callAgent(agent, task);

      task.status = 'completed';
    } catch (error) {
      task.status = 'failed';
      throw error;
    } finally {
      // Release Agent
      agent.currentTasks = agent.currentTasks.filter(id => id !== task.id);
    }
  }

  private async callAgent(agent: Agent, task: Task): Promise<void> {
    // Actually call Agent interface
    console.log(`Agent ${agent.name} executing task: ${task.name}`);
    // ... actual call logic
  }

  // Get schedule status
  getStatus(): {
    pending: number;
    running: number;
    completed: number;
    failed: number;
  } {
    const tasks = Array.from(this.tasks.values());
    return {
      pending: tasks.filter(t => t.status === 'pending').length,
      running: tasks.filter(t => t.status === 'running').length,
      completed: tasks.filter(t => t.status === 'completed').length,
      failed: tasks.filter(t => t.status === 'failed').length
    };
  }
}
```

## 5. Scheduling Optimization Strategies

### 5.1 Parallelization Optimization

```
Serial Execution:
A ──→ B ──→ C ──→ D
2h    3h    2h    1h = 8h

Parallelization Optimization:
A ──→ B ──┬──→ C ──→ D
          │
          └──→ E ──→ F
2h    3h   2h    1h = 8h (no optimization)

After Optimization:
A ──→ B ──→ C ──→ D
2h    3h    2h    1h = 8h
└──→ E ──→ F
     2h    1h = 3h (parallel)

Total time: max(8h, 2h+3h) = 8h → 5h (A+E parallel)
```

### 5.2 Priority Scheduling

```typescript
class PriorityScheduler extends TaskScheduler {
  // Priority weights
  private priorityWeights = {
    critical: 1000,
    high: 100,
    medium: 10,
    low: 1
  };

  selectBestAgent(task: Task): Agent | null {
    // Sort by priority first
    const priorityScore = this.priorityWeights[task.priority];

    // Assign among high priority tasks
    return super.selectBestAgent(task);
  }
}
```

### 5.3 Failure Retry

```typescript
class ResilientScheduler extends TaskScheduler {
  private maxRetries = 3;
  private retryDelay = 5000; // 5 seconds

  private async executeTask(task: Task, agent: Agent): Promise<void> {
    let retries = 0;

    while (retries < this.maxRetries) {
      try {
        await super.executeTask(task, agent);
        return;
      } catch (error) {
        retries++;

        if (retries >= this.maxRetries) {
          task.status = 'failed';
          throw error;
        }

        console.log(`Task ${task.name} failed, retrying in ${retries} seconds...`);
        await new Promise(resolve => setTimeout(resolve, this.retryDelay));

        // Try switching Agent
        const newAgent = this.selectBestAgent(task);
        if (newAgent && newAgent.id !== agent.id) {
          agent = newAgent;
        }
      }
    }
  }
}
```

## 6. Section Summary

### Core Concepts

1. **Task Decomposition** is the foundation of managing complex tasks, following the MECE principle

2. **Dependency Management** uses DAG to identify task execution order and critical path

3. **Task Assignment** needs to comprehensively consider capability matching, load balancing, and priority

4. **Scheduling Optimization** includes parallelization, priority scheduling, and failure retry

### Key Takeaways

- Task decomposition should be moderate - too coarse is hard to execute, too fine increases management cost
- Dependency graph helps identify parallelization opportunities and critical paths
- Dynamic scheduling adapts to changes better than static assignment
- Failure handling mechanisms ensure system robustness
