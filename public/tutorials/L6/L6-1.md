# L6-1: Multi-Agent 协作概述

> 从单打独斗到团队协作，让多个 AI Agent 共同解决复杂问题

## 本节导读

单个 AI Agent 的能力是有限的，就像一个人难以完成大型项目一样。当面对复杂任务时，我们需要**多个 Agent 协同工作**，各自发挥专长，共同完成目标。

本节课将带你了解：
- 为什么需要 Multi-Agent 协作
- Multi-Agent 系统的核心架构
- 常见的协作模式
- 如何设计有效的 Agent 团队

---

## 一、从单 Agent 到 Multi-Agent

### 1.1 单 Agent 的局限性

想象一个人要独立完成一个电商平台的开发：

```
┌─────────────────────────────────────────────────────────────┐
│                    单 Agent 的挑战                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   任务：开发电商平台                                          │
│                                                             │
│   单 Agent 尝试：                                             │
│   ├─ 需求分析（能做）✓                                       │
│   ├─ 架构设计（能做）✓                                       │
│   ├─ 前端开发（能做）✓                                       │
│   ├─ 后端开发（能做）✓                                       │
│   ├─ 数据库设计（能做）✓                                     │
│   ├─ 支付集成（了解有限）△                                   │
│   ├─ 安全审计（不够深入）△                                   │
│   ├─ 性能优化（容易遗漏）△                                   │
│   └─ 测试覆盖（难以全面）△                                   │
│                                                             │
│   结果：样样通，样样松                                        │
│   - 上下文超载，容易遗忘                                      │
│   - 专业深度不够                                              │
│   - 质量参差不齐                                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**具体问题：**
- **上下文限制**：难以记住整个项目的所有细节
- **专业深度**：对每个领域的理解都不够深入
- **一致性**：不同部分的代码风格和质量不一致
- **效率**：串行处理，耗时较长

### 1.2 Multi-Agent 的优势

```
┌─────────────────────────────────────────────────────────────┐
│                  Multi-Agent 协作                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│                    ┌─────────────┐                          │
│                    │   协调者     │                          │
│                    │  (Orchestrator)│                        │
│                    └──────┬──────┘                          │
│                           │                                 │
│        ┌──────────────────┼──────────────────┐              │
│        │                  │                  │              │
│   ┌────┴────┐        ┌────┴────┐        ┌────┴────┐        │
│   │ 架构师   │        │ 前端开发 │        │ 后端开发 │        │
│   │ Agent   │        │ Agent  │        │ Agent  │        │
│   └────┬────┘        └────┬────┘        └────┬────┘        │
│        │                  │                  │              │
│   ┌────┴────┐        ┌────┴────┐        ┌────┴────┐        │
│   │ 数据库   │        │ 安全专家 │        │ 测试工程师│        │
│   │ Agent   │        │ Agent  │        │ Agent  │        │
│   └─────────┘        └─────────┘        └─────────┘        │
│                                                             │
│   结果：专业分工，协作完成                                    │
│   - 每个 Agent 专注自己的领域                                 │
│   - 并行处理，提升效率                                        │
│   - 相互审查，保证质量                                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**核心价值：**

| 维度 | 单 Agent | Multi-Agent |
|------|----------|-------------|
| **专业深度** | 广泛但浅显 | 每个领域深入专业 |
| **任务并行** | 串行执行 | 多任务并行 |
| **质量保证** | 自我检查 | 相互审查 |
| **可扩展性** | 受限于上下文 | 可动态添加 Agent |
| **复杂度处理** | 适合简单任务 | 可处理复杂项目 |

### 2.2 协调者（Orchestrator）

**职责：**
- 接收用户请求
- 分解任务
- 分配给合适的 Agent
- 监控执行进度
- 整合最终结果

**工作流程：**

```
用户请求
    │
    ▼
┌──────────┐    ┌──────────┐    ┌──────────┐
│ 任务理解  │───→│ 任务分解  │───→│ 依赖分析  │
└──────────┘    └──────────┘    └──────────┘
                                       │
                                       ▼
                              ┌──────────┐
                              │ 制定计划  │
                              │ (DAG)    │
                              └────┬─────┘
                                   │
                    ┌──────────────┼──────────────┐
                    │              │              │
                    ▼              ▼              ▼
              ┌──────────┐  ┌──────────┐  ┌──────────┐
              │ Agent A  │  │ Agent B  │  │ Agent C  │
              │ 并行执行  │  │ 并行执行  │  │ 并行执行  │
              └────┬─────┘  └────┬─────┘  └────┬─────┘
                   │              │              │
                   └──────────────┼──────────────┘
                                  │
                                  ▼
                           ┌──────────┐
                           │ 结果整合  │
                           └────┬─────┘
                                │
                                ▼
                           ┌──────────┐
                           │ 质量检查  │
                           └────┬─────┘
                                │
                                ▼
                           ┌──────────┐
                           │ 返回用户  │
                           └──────────┘
```

**通信模式：**

```
模式 1：直接通信                    模式 2：通过消息总线
┌─────────┐      ┌─────────┐       ┌─────────┐      ┌─────────┐
│ Agent A │←────→│ Agent B │       │ Agent A │─────→│ 消息总线 │
└─────────┘      └─────────┘       └─────────┘      └────┬────┘
                                                         │
                              ┌─────────┐      ┌────────┴────────┐

│ Agent C │←──订阅── 主题 X
└─────────┘
```

## 三、常见的协作模式

### 3.1 流水线模式（Pipeline）

**适用场景：** 任务有明确的步骤顺序

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 需求分析  │───→│ 架构设计  │───→│ 代码开发  │───→│ 测试验证  │
│  Agent   │    │  Agent   │    │  Agent   │    │  Agent   │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
     │               │               │               │
     ▼               ▼               ▼               ▼
  PRD 文档       技术方案        代码实现        测试报告
```

**示例：代码生成流水线**

```yaml
workflow:
  name: code_generation_pipeline
  
  steps:
    - name: requirement_analysis
      agent: requirement_analyst
      input: user_request
      output: prd_document
    
    - name: architecture_design
      agent: architect
      input: prd_document
      output: tech_spec
    
    - name: interface_design
      agent: interface_designer
      input: tech_spec
      output: api_contract
    
    - name: implementation
      agent: developer
      input: [tech_spec, api_contract]
      output: code
    
    - name: code_review
      agent: code_reviewer
      input: code
      output: review_report
    
    - name: testing
      agent: tester
      input: [code, review_report]
      output: test_report
```

### 3.2 分治模式（Divide and Conquer）

**适用场景：** 任务可分解为独立的子任务

```
                    ┌──────────┐
                    │  协调者   │
                    └────┬─────┘
                         │
         ┌───────────────┼───────────────┐
         │               │               │
         ▼               ▼               ▼
    ┌──────────┐   ┌──────────┐   ┌──────────┐
    │ Agent A  │   │ Agent B  │   │ Agent C  │
    │ 用户模块  │   │ 商品模块  │   │ 订单模块  │
    └────┬─────┘   └────┬─────┘   └────┬─────┘
         │               │               │
         └───────────────┼───────────────┘
                         │
                         ▼
                    ┌──────────┐
                    │ 结果整合  │
                    └──────────┘
```

### 3.3 讨论模式（Discussion）

**适用场景：** 需要多轮讨论达成共识

```
┌──────────┐         ┌──────────┐         ┌──────────┐
│ 提议者   │────────→│ 评估者   │────────→│ 决策者   │
│ Agent   │         │ Agent   │         │ Agent   │
│ "我建议" │         │ "优缺点" │         │ "决定"   │
└──────────┘         └──────────┘         └──────────┘
     ↑                                        │
     └────────────────────────────────────────┘
              "反馈迭代"
```

**示例：技术方案讨论**

```
Round 1:
架构师 Agent: 提议使用微服务架构
    ↓
评估 Agent: 分析优缺点
  - 优点：可扩展性好
  - 缺点：复杂度高，运维成本大
    ↓
决策者 Agent: 建议先采用单体架构，后续再拆分
    ↓
架构师 Agent: 接受建议，修改方案

Round 2:
架构师 Agent: 提出修改后的方案
    ↓
... (直到达成共识)
```

### 3.4 竞争模式（Competition）

**适用场景：** 需要多个方案对比选择

```
                    ┌──────────┐
                    │  任务    │
                    └────┬─────┘
                         │
         ┌───────────────┼───────────────┐
         │               │               │
         ▼               ▼               ▼
    ┌──────────┐   ┌──────────┐   ┌──────────┐
    │ Agent A  │   │ Agent B  │   │ Agent C  │
    │ 方案 1   │   │ 方案 2   │   │ 方案 3   │
    └────┬─────┘   └────┬─────┘   └────┬─────┘
         │               │               │
         └───────────────┼───────────────┘
                         │
                         ▼
                    ┌──────────┐
                    │ 评估 Agent│
                    │ 选择最佳  │
                    └──────────┘
```

### 3.5 模式对比

| 模式 | 适用场景 | 优点 | 缺点 |
|------|----------|------|------|
| **流水线** | 有明确步骤的任务 | 结构清晰，易于管理 | 灵活性差，不能并行 |
| **分治** | 可分解的独立任务 | 并行高效 | 需要良好的任务划分 |
| **讨论** | 需要达成共识 | 考虑全面，质量高 | 耗时较长 |
| **竞争** | 需要多方案对比 | 方案多样，可选最优 | 资源消耗大 |

## 四、Agent 角色设计

```yaml
agents:
  # 产品经理
```

### 4.2 角色协作关系

```
┌─────────────────────────────────────────────────────────────┐
│                    Agent 协作关系图                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   产品经理 ──→ 架构师 ──→ 前端开发 ──→ 代码审查 ──→ 测试   │
│      │           │           │           │           │     │
│      └───────────┴───────────┴───────────┴───────────┘     │

│                                                             │

```

## 五、实战：设计一个开发团队

### 5.1 团队配置

**项目：** 开发一个任务管理系统

**团队构成：**

```yaml
team:
  name: "TaskFlow 开发团队"
  
  orchestrator:
    name: "项目经理"
    model: "gpt-4"
    system_prompt: |
      你是项目经理，负责协调开发团队完成项目。
      你的职责：
      1. 理解用户需求
      2. 分解任务并分配给合适的 Agent
      3. 监控进度，处理阻塞
      4. 整合各 Agent 的输出
      5. 确保最终交付质量
  
  agents:
    - name: "需求分析师"
      role: "分析用户需求，编写 PRD"
      model: "gpt-4"
      knowledge:
        - product-management
        - user-story-writing
      
    - name: "架构师"
      role: "设计系统架构"
      model: "gpt-4"
      knowledge:
        - system-design
        - database-design
        - api-design
      
    - name: "前端开发"
      role: "实现前端功能"
      model: "gpt-4"
      knowledge:
        - react
        - typescript
        - ui-design
      tools:
        - file_system
        - code_generator
      
    - name: "后端开发"
      role: "实现后端功能"
      model: "gpt-4"
      knowledge:
        - nodejs
        - database
        - rest-api
      tools:
        - file_system
        - code_generator
      
    - name: "代码审查员"
      role: "审查代码质量"
      model: "gpt-4"
      knowledge:
        - code-review
        - security
        - performance
      
    - name: "测试工程师"
      role: "编写和执行测试"
      model: "gpt-4"
      knowledge:
        - testing
        - jest
        - cypress
```

### 5.2 工作流程定义

```yaml
workflow:
  name: "TaskFlow 开发流程"
  
  triggers:
    - user_request
  
  steps:
    # 阶段 1：需求分析
    - id: analyze_requirements
      name: "需求分析"
      agent: "需求分析师"
      input: user_request
      output: prd_document
      
    # 阶段 2：架构设计
    - id: design_architecture
      name: "架构设计"
      agent: "架构师"
      input: prd_document
      output: architecture_document
      depends_on: [analyze_requirements]
      
    # 阶段 3：并行开发
    - id: develop_frontend
      name: "前端开发"
      agent: "前端开发"
      input: [prd_document, architecture_document]
      output: frontend_code
      depends_on: [design_architecture]
      
    - id: develop_backend
      name: "后端开发"
      agent: "后端开发"
      input: [prd_document, architecture_document]
      output: backend_code
      depends_on: [design_architecture]
      
    # 阶段 4：代码审查（并行审查）
    - id: review_frontend
      name: "前端代码审查"
      agent: "代码审查员"
      input: frontend_code
      output: frontend_review
      depends_on: [develop_frontend]
      
    - id: review_backend
      name: "后端代码审查"
      agent: "代码审查员"
      input: backend_code
      output: backend_review
      depends_on: [develop_backend]
      
    # 阶段 5：修复问题（如果需要）
    - id: fix_frontend
      name: "前端问题修复"
      agent: "前端开发"
      input: [frontend_code, frontend_review]
      output: frontend_code_v2
      depends_on: [review_frontend]
      condition: "has_issues(frontend_review)"
      
    - id: fix_backend
      name: "后端问题修复"
      agent: "后端开发"
      input: [backend_code, backend_review]
      output: backend_code_v2
      depends_on: [review_backend]
      condition: "has_issues(backend_review)"
      
    # 阶段 6：测试
    - id: run_tests
      name: "执行测试"
      agent: "测试工程师"
      input: [frontend_code_v2, backend_code_v2]
      output: test_report
      depends_on: [fix_frontend, fix_backend]
      
    # 阶段 7：最终交付
    - id: deliver
      name: "项目交付"
      agent: "项目经理"
      input: [prd_document, architecture_document, frontend_code_v2, backend_code_v2, test_report]
      output: delivery_package
      depends_on: [run_tests]
```

## 六、挑战与解决方案

### 6.1 常见挑战

| 挑战 | 描述 | 解决方案 |
|------|------|----------|
| **上下文同步** | Agent 之间信息不一致 | 共享知识库，统一状态管理 |
| **冲突解决** | 多个 Agent 意见不一致 | 投票机制，协调者裁决 |
| **循环依赖** | Agent A 等 B，B 等 A | 依赖图检测，打破循环 |
| **质量不一** | 不同 Agent 输出质量不同 | 审查机制，质量标准 |
| **成本过高** | 多个 Agent 调用费用高 | 任务合并，选择性使用 |

### 6.2 最佳实践

1. **明确定义角色边界**
   - 每个 Agent 有清晰的职责范围
   - 避免职责重叠

2. **建立标准化接口**
   - 统一的输入输出格式
   - 规范化的消息协议

3. **实施质量门禁**
   - 每个阶段的输出必须经过检查
   - 不合格返回重新处理

4. **记录完整日志**
   - 追踪每个决策的原因
   - 便于问题排查和优化

5. **支持人工介入**
   - 关键决策需要人工确认
   - 异常情况人工处理

## 七、本节小结

### 核心概念

1. **Multi-Agent 协作** 让多个专业 Agent 共同完成复杂任务

2. **核心架构**：协调层 + 通信层 + Agent 层 + 共享层

3. **协作模式**：流水线、分治、讨论、竞争，根据场景选择

4. **角色设计**：明确定义每个 Agent 的职责、技能和输出

### 关键要点

- 单 Agent 有上下文和专业深度限制
- Multi-Agent 通过分工协作解决复杂问题
- 协调者是系统的核心，负责任务分配和进度监控
- 选择合适的协作模式很重要
