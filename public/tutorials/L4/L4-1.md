# 结构化提示词



## 为什么需要结构化提示词？

想象一下，如果你让一个人做事，你会怎么说？

> ❌ **不好的方式**：
> "去做个东西"
> 
> **结果**：对方完全不知道要做什么

---

> ✅ **好的方式**：
> 
> "你是一名厨师（角色），
>  现在需要为 5 个人准备晚餐（背景），
>  请做三道菜：一道荤菜、一道素菜、一道汤（任务），
>  要求：不辣、30 分钟内完成、预算 100 元（约束）"
> 
> **结果**：对方清楚知道要做什么

**结构化提示词就是让 AI 清楚知道你要什么。**

## 结构化提示词的核心框架

### RCTC 框架

| 要素 | 英文 | 说明 | 示例 |
|------|------|------|------|
| **R**ole | 角色 | AI 扮演什么角色 | 你是一名资深前端工程师 |
| **C**ontext | 背景 | 相关的背景信息 | 我们正在开发一个电商网站 |
| **T**ask | 任务 | 要完成什么任务 | 请实现购物车功能 |
| **C**onstraints | 约束 | 有什么限制条件 | 使用 React，支持 TypeScript |

### 扩展框架：RCTC + E

在 RCTC 基础上增加 **E**xamples（示例）：

| 要素 | 说明 |
|------|------|
| **R**ole | 角色 |
| **C**ontext | 背景 |
| **T**ask | 任务 |
| **C**onstraints | 约束 |
| **E**xamples | 示例 |

## 每个要素详解

### 1. Role（角色）

**作用**：让 AI 以特定身份回答问题，获得更专业的输出。

**示例**：

> ❌ **无角色**：
> "解释一下 React"

---

> ✅ **有角色**：
> "你是一名有 10 年经验的前端架构师，
>  请向一名初级开发者解释 React 的核心概念"
> 
> **结果**：解释更深入、更专业、更适合初学者

**常用角色**：
- 技术专家：资深工程师、架构师、技术总监
- 教育工作者：老师、导师、培训师
- 业务专家：产品经理、设计师、运营
- 审核者：代码审查员、测试工程师

### 2. Context（背景）

**作用**：提供必要的上下文，让 AI 理解问题的来龙去脉。

**示例**：

> ❌ **无背景**：
> "优化这段代码"

---

> ✅ **有背景**：
> "我们正在开发一个高并发的电商系统，
>  这段代码是商品搜索接口，
>  目前响应时间超过 2 秒，需要优化到 200ms 以内"
> 
> **结果**：AI 会给出针对性的优化方案

**背景信息包括**：
- 项目类型和规模
- 技术栈和版本
- 目标用户和场景
- 当前问题和痛点
- 已尝试的解决方案

### 3. Task（任务）

**作用**：明确告诉 AI 要做什么。

**示例**：
```markdown
❌ 模糊任务：
"处理一下这个数据"

✅ 明确任务：
"请对这个销售数据进行分析：
 1. 计算每个月的总销售额
 2. 找出销售额最高的产品
 3. 分析销售趋势
 4. 用图表展示结果"

结果：AI 知道具体要做什么
```

**任务描述技巧**：
- 使用动词开头：实现、分析、优化、创建...
- 分解为具体步骤
- 明确输出格式

### 4. Constraints（约束）

**作用**：限定 AI 的输出范围，避免不符合要求的结果。

**示例**：
```markdown
❌ 无约束：
"写一个登录功能"

✅ 有约束：
"写一个登录功能，要求：
 - 使用 JWT 认证
 - 密码使用 bcrypt 加密
 - 支持记住我功能（7天）
 - 响应时间 < 100ms
 - 代码要有完整注释"

结果：AI 会按照约束生成代码
```

**常见约束类型**：
- 技术约束：语言、框架、库版本
- 性能约束：响应时间、内存占用
- 安全约束：加密方式、权限控制
- 格式约束：代码风格、文档格式
- 时间约束：完成时间、截止日期

### 5. Examples（示例）

**作用**：通过示例说明期望的输出格式和质量。

**示例**：
```markdown
无示例：
"将以下文本分类"

有示例：
"请将以下文本分类为：积极、消极、中性

示例：
文本：'这个产品太棒了！'
分类：积极

文本：'质量很差，不推荐'
分类：消极

文本：'今天天气不错'
分类：中性

现在请分类：
文本：'物流很快，包装完好'
分类：？"

结果：AI 知道分类标准和格式
```

## 完整示例

### 示例 1：代码审查

```markdown
## Role
你是一名资深的前端代码审查员，有 10 年 React 开发经验。

## Context
我们正在开发一个企业级后台管理系统，使用 React 18 + TypeScript。
这是用户管理页面的代码，需要确保代码质量和性能。

## Task
请审查以下代码，找出潜在问题：

```tsx
import React, { useState, useEffect } from 'react';

function UserList() {
  const [users, setUsers] = useState([]);
  
  useEffect(() => {
    fetch('/api/users')
      .then(res => res.json())
      .then(data => setUsers(data));
  });
  
  return (
    <div>
      {users.map(user => (
        <div key={user.id}>{user.name}</div>
      ))}
    </div>
  );
}
```

## Constraints
1. 重点关注：性能问题、内存泄漏、类型安全
2. 每个问题给出：严重程度、具体位置、修复建议
3. 提供优化后的完整代码
```

### 示例 2：功能实现

```markdown
## Role
你是一名全栈工程师，擅长 Node.js 和 PostgreSQL。

## Context
我们正在开发一个博客系统，需要实现文章评论功能。
技术栈：Node.js + Express + PostgreSQL + TypeScript

## Task
请实现评论功能的 API 接口，包括：
1. 发表评论
2. 获取评论列表（支持分页）
3. 删除评论（仅作者或管理员）

## Constraints
- 使用 RESTful API 设计
- 需要输入验证（使用 zod）
- 需要错误处理
- 代码要有完整注释
- 包含单元测试

## Examples
输入验证示例：
```typescript
const createCommentSchema = z.object({
  postId: z.string().uuid(),
  content: z.string().min(1).max(1000),
  parentId: z.string().uuid().optional()
});
```

### 示例 3：文档编写

#### Role
你是一名技术文档工程师，擅长编写清晰易懂的技术文档。

#### Context
我们开发了一个 React 组件库，需要为 Button 组件编写使用文档。
目标读者：初中级前端开发者

#### Task
请编写 Button 组件的文档，包括：
1. 组件介绍
2. Props 说明
3. 使用示例
4. 常见问题

#### Constraints
- 使用 Markdown 格式
- 包含代码示例
- 语言简洁明了
- 适合快速上手

#### Examples
Props 表格格式：
| 属性 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| size | 'sm' \| 'md' \| 'lg' | 'md' | 按钮尺寸 |

## 常见错误

### 错误 1：缺少角色

> ❌ "解释什么是闭包"
>
> ✅ "你是一名 JavaScript 专家，请向初学者解释什么是闭包，用生活中的例子类比"

### 错误 2：背景不足

> ❌ "优化这段代码"
>
> ✅ "这段代码是用户列表页面，目前有 10000+ 用户，页面加载很慢，请优化性能"

### 错误 3：任务模糊

> ❌ "处理一下这个需求"
>
> ✅ "请分析这个需求，输出：
>      1. 功能模块划分
>      2. 数据模型设计
>      3. 接口定义"

### 错误 4：约束不清

> ❌ "写一个登录功能"
>
> ✅ "写一个登录功能，要求：
>      - 使用邮箱+密码
>      - JWT Token 认证
>      - Token 有效期 24 小时
>      - 错误提示友好"
