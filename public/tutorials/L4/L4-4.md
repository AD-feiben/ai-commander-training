# 迭代式提示优化



## 什么是迭代式提示？

**迭代式提示**是指通过多轮对话，逐步优化 AI 的输出。就像打磨石头一样，每一轮都让结果更好一点。

> 不要期望一次就完美，迭代是常态。

## 为什么需要迭代？

### AI 一次生成的局限性

- **理解偏差**：AI 可能误解你的意图
- **细节遗漏**：复杂任务容易遗漏细节
- **质量不均**：某些部分好，某些部分差
- **边界情况**：容易忽略边界条件

### 迭代的好处

- **逐步完善**：从 60 分到 90 分
- **及时纠错**：发现问题立即修正
- **深度优化**：针对特定方面改进
- **符合预期**：最终结果符合你的要求

## 迭代的基本流程

```
第一轮：生成初版
    ↓
第二轮：指出问题 → AI 修正
    ↓
第三轮：优化细节 → AI 改进
    ↓
第四轮：最终确认 → 完成
```

## 实战示例

### 示例 1：代码生成迭代

**第一轮：生成初版**

```
你：帮我写一个邮箱验证函数

AI：
```javascript
function validateEmail(email) {
  return email.includes('@');
}
```

**第二轮：指出问题**

```
你：这个验证太简单了，需要更严格的验证：
1. 验证邮箱格式（正则表达式）
2. 验证长度（最长 254 字符）
3. 验证域名部分

AI：
```javascript
function validateEmail(email) {
  if (!email || email.length > 254) {
    return false;
  }
  const regex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
  return regex.test(email);
}
```

**第三轮：优化细节**

```
你：很好！再优化一下：
1. 添加 JSDoc 注释
2. 返回具体的错误信息，不只是 true/false
3. 处理空字符串和 null

AI：
```javascript
/**
 * 验证邮箱地址
 * @param email - 要验证的邮箱
 * @returns 验证结果对象
 */
function validateEmail(email) {
  // 检查空值
  if (!email) {
    return { valid: false, error: '邮箱不能为空' };
  }
  
  // 检查类型
  if (typeof email !== 'string') {
    return { valid: false, error: '邮箱必须是字符串' };
  }
  
  // 检查长度
  if (email.length > 254) {
    return { valid: false, error: '邮箱长度不能超过 254 字符' };
  }
  
  // 检查格式
  const regex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
  if (!regex.test(email)) {
    return { valid: false, error: '邮箱格式不正确' };
  }
  
  return { valid: true, error: null };
}
```

**第四轮：最终优化**

```
你：最后优化一下：
1. 使用 TypeScript
2. 添加单元测试

AI：
```typescript
interface ValidationResult {
  valid: boolean;
  error: string | null;
}

/**
 * 验证邮箱地址
 * @param email - 要验证的邮箱
 * @returns 验证结果对象
 */
function validateEmail(email: unknown): ValidationResult {
  if (!email) {
    return { valid: false, error: '邮箱不能为空' };
  }
  
  if (typeof email !== 'string') {
    return { valid: false, error: '邮箱必须是字符串' };
  }
  
  if (email.length > 254) {
    return { valid: false, error: '邮箱长度不能超过 254 字符' };
  }
  
  const regex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
  if (!regex.test(email)) {
    return { valid: false, error: '邮箱格式不正确' };
  }
  
  return { valid: true, error: null };
}

// 单元测试
describe('validateEmail', () => {
  test('valid email', () => {
    expect(validateEmail('test@example.com')).toEqual({
      valid: true,
      error: null
    });
  });
  
  test('empty email', () => {
    expect(validateEmail('')).toEqual({
      valid: false,
      error: '邮箱不能为空'
    });
  });
  
  test('invalid format', () => {
    expect(validateEmail('invalid-email')).toEqual({
      valid: false,
      error: '邮箱格式不正确'
    });
  });
});
```

## 迭代技巧

### 技巧 1：具体反馈

❌ 不好的反馈：
```
"这段代码不好"
```

✅ 好的反馈：
```
"这段代码在处理边界情况时有问题：
- 当输入为 null 时会报错
- 没有验证参数类型
- 建议添加错误处理"
```

### 技巧 2：一次改一点

❌ 一次提太多要求：
```
"请修改：添加类型、优化性能、添加注释、
修复 bug、添加测试、改进命名..."
```

✅ 一次一个重点：
```
第一轮："请添加 TypeScript 类型"
第二轮："请修复边界情况的处理"
第三轮："请添加单元测试"
```

### 技巧 3：明确要求

❌ 说不要什么：
```
"不要写得太复杂"
```

✅ 说要什么：
```
"请保持简单，只保留核心功能，
去掉不必要的抽象"
```

### 技巧 4：提供参考

```
"请参考这个风格修改：
[粘贴参考代码]

请按照这个格式重写你的代码"
```

## 迭代模式

### 模式 1：纠错式迭代

发现问题 → 指出问题 → AI 修正

```
你：这段代码有 bug，当数组为空时会出错
AI：已修复，添加了空数组检查
```

### 模式 2：完善式迭代

基础版本 → 补充细节 → 添加功能

```
你：请添加错误处理
你：请添加日志记录
你：请添加性能监控
```

### 模式 3：优化式迭代

能运行 → 代码整洁 → 性能优化

```
你：请重构代码，提高可读性
你：请优化算法，提高性能
```

## 常见错误

### 错误 1：期望一次完美

```
❌ 第一轮就提所有要求
结果：AI 无法同时处理，质量下降

✅ 分轮次逐步完善
结果：每轮都有改进，最终质量高
```

### 错误 2：反馈模糊

```
❌ "感觉不太对"
结果：AI 不知道改什么

✅ "第 3 行的逻辑有问题，应该..."
结果：AI 能准确修改
```

### 错误 3：频繁改需求

```
❌ 每轮都改需求方向
结果：永远无法完成

✅ 保持需求稳定，逐步完善
结果：高效完成
```

## 迭代检查清单

每轮迭代后检查：

- [ ] 是否解决了上一轮的问题
- [ ] 是否引入了新的问题
- [ ] 是否更接近预期结果
- [ ] 是否还有改进空间
