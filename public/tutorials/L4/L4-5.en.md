## I. Core Principles of Prompt Optimization

### 1.1 Why Do We Need Prompt Optimization?

**Case Comparison: Same Requirements, Different Results**

❌ **Unoptimized prompt:**
```
Write a login function
```

AI might return a simple HTML form, lacking backend validation, security measures, and user experience considerations.

✅ **Optimized prompt:**
```markdown
## Role
You are a full-stack secure development expert, proficient in building enterprise-level authentication systems.

## Task
Design and implement a complete user login system.

## Technical Requirements
- Frontend: React + TypeScript + React Hook Form
- Backend: Node.js + Express + JWT
- Database: PostgreSQL + bcrypt password hashing
- Security: CSRF protection, rate limiting, SQL injection protection

## Functional Requirements
1. Email/password login
2. Form validation (frontend + backend dual validation)
3. JWT Token management (Access + Refresh Token)
4. Login failure handling (error messages, account lockout strategy)
5. Remember me function (7-day validity)

## Output Requirements
1. Complete frontend component code
2. Backend API interface code
3. Database table structure
4. Security considerations checklist
```

This prompt will yield a complete, secure, production-ready login system implementation.

**Five Golden Rules:**

| Rule | Description | Example |
|------|-------------|---------|
| **Be Specific** | Avoid vague words, use precise descriptions | ❌ "Optimize code" → ✅ "Reduce time complexity from O(n²) to O(n log n)" |
| **Layered Progression** | From basic to advanced, gradually refine | First framework → Then details → Finally edge cases |
| **Example-Driven** | Use examples to illustrate expected output | Provide input/output example pairs |
| **Clear Constraints** | Specify tech stack, standards, limitations | "Use TypeScript, follow Airbnb style guide" |
| **Verifiable** | Output results can be tested and verified | Include test cases or acceptance criteria |

## II. Scenario-Based Prompt Design Patterns

### 2.1 Code Generation Scenario

**Applicable scenario:** Need AI to generate complete functional code

**Template structure:**
```markdown
## Role Definition
You are a [tech stack] expert, proficient in [related technologies].

## Task Description
Implement [specific feature] for [business scenario].

## Technical Constraints
- Language/Framework: [specific tech stack]
- Code Standards: [standards to follow]
- Performance Requirements: [time/space complexity]
- Compatibility Requirements: [browser/Node version]

## Input Definition
[Describe input data structure, types, ranges]

## Output Definition
[Describe expected output format and content]

## Examples
### Example 1: Normal case
Input: [specific input]
Output: [expected output]

### Example 2: Edge case
Input: [edge input]
Output: [expected handling]

## Quality Requirements
- [ ] Code passes all example tests
- [ ] Includes error handling logic
- [ ] Has clear comments
- [ ] Includes unit test code
```

**Practical example: Data validation function generation**

```markdown
## Role Definition
You are a TypeScript expert, proficient in building type-safe validation libraries.

## Task Description
Implement a user registration data validation function to validate email, password, and username.

## Technical Constraints
- Language: TypeScript 5.0+
- Use Zod library for schema validation
- Error messages in English
- Support custom error messages

## Input Definition
```typescript
interface RegisterData {
  email: string;      // Email format
  password: string;   // 8-32 characters, containing uppercase, lowercase, and numbers
  username: string;   // 3-20 characters, starting with letter, only letters, numbers, underscores
  confirmPassword: string; // Must match password
}
```

## Output Definition
Return validation result object:
```typescript
{
  success: boolean;
  data?: RegisterData;      // Return cleaned data when validation passes
  errors?: Record<string, string>; // Return field error messages when validation fails
}
```

## Examples
### Example 1: Validation passes
Input: { email: "user@example.com", password: "Pass1234", username: "john_doe", confirmPassword: "Pass1234" }
Output: { success: true, data: {...} }

### Example 2: Validation fails
Input: { email: "invalid", password: "123", username: "ab", confirmPassword: "456" }
Output: {
  success: false,
  errors: {
    email: "Please enter a valid email address",
    password: "Password must contain uppercase, lowercase letters and numbers, 8-32 characters",
    username: "Username length must be between 3-20 characters",
    confirmPassword: "Passwords do not match"
  }
}

## Quality Requirements
- [ ] Use Zod to define schema
- [ ] Each field has independent error messages
- [ ] Support async validation (e.g., check if email is already registered)
- [ ] Include more than 5 unit test cases
```

**Applicable scenario:** Need AI to check code quality and discover potential issues

**Template structure:**

```markdown

- **Fix suggestion**: Specific code modification plan

```

## Special Notes

```

**Practical example: API interface security review**

```markdown
## Role Definition
You are a web application security expert, proficient in OWASP Top 10 security vulnerability detection.

## Review Objectives
Conduct a security review of the following user registration API, focusing on:
1. Input validation and injection attack prevention
2. Sensitive data handling
3. Authentication and authorization
4. Rate limiting and brute force protection

## Code Context
This is a Node.js + Express user registration endpoint, using MongoDB to store user data.

## Code to Review
```javascript
app.post('/api/register', async (req, res) => {
  const { username, email, password } = req.body;

  const user = new User({
    username,
    email,
    password
  });

  await user.save();

  res.json({
    success: true,
    message: 'Registration successful',
    user: user
  });
});
```

## Review Dimensions
- 🔴 Critical: SQL/NoSQL injection, sensitive data leakage, permission bypass
- 🟡 Warning: Lack of input validation, error message leakage, weak password policy
- 🟢 Suggestion: Logging, code structure, exception handling
- 💡 Optimization: Performance, scalability

## Special Notes
1. Check for NoSQL injection vulnerabilities
2. Confirm if password is stored in plaintext
3. Check if sensitive fields (like password hash) are leaked in response
4. Verify if there is replay attack protection mechanism
```

**Practical example: React component rendering exception diagnosis**

```markdown
## Problem Phenomenon
React component enters an infinite loop during rendering, causing the page to freeze.
Browser console displays "Maximum update depth exceeded" error.

## Environment Information
- Operating System: macOS 14.0
- Programming Language: TypeScript 5.2
- Framework: React 18.2
- Runtime Environment: Chrome 120

## Reproduction Steps
1. Open user list page
2. Click filter button and select "Active users"
3. Page freezes, console reports error

## Related Code
```typescript
const UserList: React.FC = () => {
  const [users, setUsers] = useState<User[]>([]);
  const [filter, setFilter] = useState<string>('all');

  useEffect(() => {
    fetchUsers(filter).then(data => {
      setUsers(data);
      setFilter('all'); // Reset filter
    });
  }, [filter]);

  return (
    <div>
      <Filter onChange={setFilter} value={filter} />
      <UserTable users={users} />
    </div>
  );
};
```

## Error Information
```
Warning: Maximum update depth exceeded.
This can happen when a component calls setState inside useEffect
without proper dependencies.
```

## Attempted Solutions
1. Changed useEffect dependencies to empty array [] - Ineffective, data doesn't update
2. Added loading state - Problem still exists

## Expected Behavior
After filtering users, display the corresponding list without triggering infinite loop.

## Diagnosis Requirements
Please analyze:
1. Why does the infinite loop occur?
2. How to fix this problem?
3. What is a better state management solution?
```

## III. Prompt Effectiveness Evaluation and Measurement

### 3.1 How to Evaluate Prompt Quality?

**Evaluation dimensions:**

| Dimension | Weight | Evaluation Criteria |
|-----------|--------|---------------------|
| **Accuracy** | 30% | Whether output is correct and meets expectations |
| **Completeness** | 25% | Whether all requirements are covered |
| **Usability** | 20% | Whether output can be used directly or with minor modifications |
| **Consistency** | 15% | Whether multiple runs produce stable consistent results |
| **Efficiency** | 10% | Whether concise and efficient, no redundant information |

**Scoring example:**
```
Prompt: Implement an array deduplication function

Evaluation results:
├─ Accuracy: ★★★★★ (5/5)  - Code correctly implements deduplication
├─ Completeness: ★★★★☆ (4/5)  - Missing handling of special types (NaN, objects)
├─ Usability: ★★★★★ (5/5)  - Code can be used directly
├─ Consistency: ★★★★★ (5/5)  - Multiple runs produce consistent results
└─ Efficiency: ★★★★☆ (4/5)   - Code is concise but can be optimized to O(n)

Overall score: 4.6/5.0
```

### 3.2 A/B Testing Prompts

**Method:** Use different prompts for the same task and compare effects

**Test case: Generate API documentation**

**Prompt A (basic version):**
```
Write documentation for this function
function calculateTotal(price, quantity, discount) {
  return price * quantity * (1 - discount);
}
```

**Prompt B (optimized version):**
```markdown
## Task
Generate JSDoc format documentation for the following function.

## Requirements
1. Explain function purpose
2. Annotate type and description of each parameter
3. Provide usage examples
4. Explain return value

## Code
```javascript
function calculateTotal(price, quantity, discount) {
  return price * quantity * (1 - discount);
}
```

**Comparison results:**

| Metric | Prompt A | Prompt B |
|--------|----------|----------|
| Documentation completeness | Parameter list only | Includes description, examples, return value |
| Format standardization | Inconsistent | Standard JSDoc |
| Usability | Needs modification | Can be used directly |
| Includes examples | No | Yes |

**Conclusion: Prompt B is more effective**

│ Write Prompt │ → │   Test Verify   │ → │  Evaluate Effect    │

       ↑                                         │
       └─────────────────────────────────────────┘

**Optimization checklist:**

- [ ] Is the prompt specific enough without ambiguity?
- [ ] Is sufficient context provided?
- [ ] Are constraints clear and definite?
- [ ] Does the output format meet requirements?
- [ ] Do examples illustrate the expected output?
- [ ] Are edge cases considered?
- [ ] Do multiple runs produce consistent results?
- [ ] Can the output be used directly?

## VI. Common Mistakes and Pitfall Guide

### 6.1 The 10 Major Pitfalls in Prompt Writing

| Pitfall | Wrong Example | Correct Approach |
|---------|--------------|------------------|
| **Too vague** | "Write some good code" | "Reduce time complexity from O(n²) to O(n)" |
| **Lack of context** | Just paste code | First explain business scenario and tech stack |
| **Unclear constraints** | "Use React to write" | "Use React 18 + TypeScript + Hooks" |
| **Insufficient examples** | No input/output examples | Provide 2-3 typical examples |
| **Ignoring edge cases** | Only consider normal cases | Explicitly state exception handling requirements |
| **Format chaos** | Plain text pile-up | Use structured Markdown |
| **Too many requirements** | Request 10 features at once | Break into steps, 2-3 features at a time |
| **Ignoring verification** | No test requirements | Explicitly require unit tests |
| **Unclear role** | No role setting | Clearly define expert role positioning |
| **No iterative optimization** | One-shot completion | Continuously improve based on results |

### 6.2 Problem-Solving Checklist

**When AI output doesn't meet expectations, check:**

- [ ] Is the prompt too long? (Try simplifying)
- [ ] Are there contradictory requirements?
- [ ] Are examples clear enough?
- [ ] Is key context missing?
- [ ] Is the output format requirement clear?
- [ ] Are you using terms AI isn't familiar with?
- [ ] Is the task too complex? (Try decomposing)
- [ ] Is sufficient background information provided?
