## 一、提示词优化的核心原则

### 1.1 为什么需要优化提示词？

**案例对比：同样的需求，不同的结果**

❌ **未经优化的提示词：**
```
写一个登录功能
```

AI 可能返回一个简单的 HTML 表单，缺乏后端验证、安全措施和用户体验考虑。

✅ **优化后的提示词：**
```markdown
## 角色
你是全栈安全开发专家，擅长构建企业级身份认证系统。

## 任务
设计并实现一个完整的用户登录系统。

## 技术要求
- 前端：React + TypeScript + React Hook Form
- 后端：Node.js + Express + JWT
- 数据库：PostgreSQL + bcrypt 密码哈希
- 安全：CSRF 防护、速率限制、SQL 注入防护

## 功能需求
1. 邮箱/密码登录
2. 表单验证（前端+后端双重验证）
3. JWT Token 管理（Access + Refresh Token）
4. 登录失败处理（错误提示、账户锁定策略）
5. 记住我功能（7天有效期）

## 输出要求
1. 完整的前端组件代码
2. 后端 API 接口代码
3. 数据库表结构
4. 安全注意事项清单
```

这个提示词会得到一个完整、安全、可生产的登录系统实现。

**实战示例：数据验证函数生成**

```markdown
## 角色定位
你是 TypeScript 专家，擅长构建类型安全的验证库。

## 任务描述
实现一个用户注册数据的验证函数，验证邮箱、密码、用户名。

## 技术约束
- 语言：TypeScript 5.0+
- 使用 Zod 库进行 schema 验证
- 错误信息需要中文提示
- 支持自定义错误消息

## 输入定义
```typescript
interface RegisterData {
  email: string;      // 邮箱格式
  password: string;   // 8-32位，包含大小写字母和数字
  username: string;   // 3-20位，字母开头，只允许字母数字下划线
  confirmPassword: string; // 必须与 password 一致
}
```

## 输出定义
返回验证结果对象：
```typescript
{
  success: boolean;
  data?: RegisterData;      // 验证通过时返回清洗后的数据
  errors?: Record<string, string>; // 验证失败时返回字段错误信息
}
```

## 示例
### 示例 1：验证通过
输入：{ email: "user@example.com", password: "Pass1234", username: "john_doe", confirmPassword: "Pass1234" }
输出：{ success: true, data: {...} }

### 示例 2：验证失败
输入：{ email: "invalid", password: "123", username: "ab", confirmPassword: "456" }
输出：{ 
  success: false, 
  errors: {
    email: "请输入有效的邮箱地址",
    password: "密码必须包含大小写字母和数字，且长度8-32位",
    username: "用户名长度必须在3-20位之间",
    confirmPassword: "两次输入的密码不一致"
  }
}

## 质量要求
- [ ] 使用 Zod 定义 schema
- [ ] 每个字段有独立的错误提示
- [ ] 支持异步验证（如检查邮箱是否已注册）
- [ ] 包含 5 个以上的单元测试用例
```

## 三、提示词效果评估与度量

### 3.1 如何评估提示词质量？

**评估维度：**

| 维度 | 权重 | 评估标准 |
|------|------|----------|
| **准确性** | 30% | 输出结果是否正确、符合预期 |
| **完整性** | 25% | 是否覆盖所有需求点 |
| **可用性** | 20% | 输出是否可以直接使用或稍作修改即可使用 |
| **一致性** | 15% | 多次运行结果是否稳定一致 |
| **效率** | 10% | 是否简洁高效，无冗余信息 |

**评分示例：**
```
提示词：实现一个数组去重函数

评估结果：
├─ 准确性：★★★★★ (5/5)  - 代码正确实现去重功能
├─ 完整性：★★★★☆ (4/5)  - 缺少对特殊类型（NaN、对象）的处理
├─ 可用性：★★★★★ (5/5)  - 代码可直接使用
├─ 一致性：★★★★★ (5/5)  - 多次运行结果一致
└─ 效率：★★★★☆ (4/5)   - 代码简洁，但可优化为 O(n)

综合评分：4.6/5.0
```

### 3.2 A/B 测试提示词

**方法：** 对同一任务使用不同提示词，对比效果

**测试案例：生成 API 文档**

**提示词 A（基础版）：**
```
给这个函数写文档
function calculateTotal(price, quantity, discount) {
  return price * quantity * (1 - discount);
}
```

**提示词 B（优化版）：**
```markdown
## 任务
为以下函数生成 JSDoc 格式的文档。

## 要求
1. 说明函数用途
2. 标注每个参数的类型和说明
3. 提供使用示例
4. 说明返回值

## 代码
```javascript
function calculateTotal(price, quantity, discount) {
  return price * quantity * (1 - discount);
}
```

**对比结果：**

| 指标 | 提示词 A | 提示词 B |
|------|----------|----------|
| 文档完整性 | 仅参数列表 | 包含描述、示例、返回值 |
| 格式规范 | 不统一 | 标准 JSDoc |
| 可用性 | 需要修改 | 可直接使用 |
| 包含示例 | 否 | 是 |

**结论：提示词 B 效果更优**

## 五、实战演练：从零优化一个提示词

### 5.1 初始版本（问题诊断）

**原始提示词：**
```
帮我看看这段代码为什么报错
```

**问题分析：**
- ❌ 没有提供代码
- ❌ 没有说明错误信息
- ❌ 没有环境信息
- ❌ 没有复现步骤

**AI 可能的回复：**
```
请提供具体的代码和错误信息，我才能帮你诊断问题。
```

### 5.2 第一次优化

**优化后：**
```
这段 React 代码报错，帮我看看为什么：

function Counter() {
  const [count, setCount] = useState(0);
  
  useEffect(() => {
    setCount(count + 1);
  });
  
  return <div>{count}</div>;
}

错误：Too many re-renders
```

**改进点：**
- ✅ 提供了代码
- ✅ 说明了错误信息
- ✅ 说明了技术栈

**仍有问题：**
- ❌ 没有环境版本信息
- ❌ 没有说明期望行为
- ❌ 没有复现步骤

### 5.3 第二次优化

**优化后：**
```markdown
## 问题
React 组件出现无限循环渲染，页面卡死后报错。

## 环境
- React 18.2
- TypeScript 5.0
- Chrome 浏览器

## 代码
```typescript
import React, { useState, useEffect } from 'react';

function Counter() {
  const [count, setCount] = useState(0);
  
  useEffect(() => {
    setCount(count + 1);
  });
  
  return <div>Count: {count}</div>;
}
```

## 错误信息
```
Error: Too many re-renders. React limits the number of renders 
to prevent an infinite loop.
```

## 复现步骤
1. 渲染 Counter 组件
2. 页面显示 Count 后卡死
3. 控制台报错

## 期望行为
组件渲染后 count 为 1，不再更新。

## 诊断要求
1. 解释为什么会无限循环
2. 提供修复方案
3. 说明最佳实践
```

**效果评估：**
- ✅ 问题描述清晰
- ✅ 环境信息完整
- ✅ 代码可运行
- ✅ 错误信息准确
- ✅ 期望明确
- ✅ 诊断要求具体

**AI 可以给出：**
1. 准确的根因分析（useEffect 依赖缺失导致循环）
2. 多种修复方案（添加依赖、使用函数式更新）
3. 最佳实践建议（何时使用 useEffect）

## 六、常见错误与避坑指南

### 6.1 提示词编写的 10 大误区

| 误区 | 错误示例 | 正确做法 |
|------|----------|----------|
| **过于模糊** | "写个好点的代码" | "将时间复杂度从 O(n²) 优化到 O(n)" |
| **缺乏上下文** | 直接贴代码 | 先说明业务场景和技术栈 |
| **约束不清** | "用 React 写" | "使用 React 18 + TypeScript + Hooks" |
| **示例不足** | 不提供输入/输出示例 | 提供 2-3 个典型示例 |
| **忽视边界** | 只考虑正常情况 | 明确说明异常处理要求 |
| **格式混乱** | 纯文本堆砌 | 使用结构化 Markdown |
| **要求过多** | 一次要求 10 个功能 | 分步骤，一次 2-3 个功能 |
| **忽视验证** | 不要求测试 | 明确要求单元测试 |
| **角色不明** | 不设定角色 | 明确专家角色定位 |
| **不迭代优化** | 一次成型 | 根据结果持续改进 |

### 6.2 问题排查清单

**当 AI 输出不符合预期时，检查：**

- [ ] 提示词是否太长？（尝试精简）
- [ ] 是否包含矛盾的要求？
- [ ] 示例是否足够清晰？
- [ ] 是否遗漏了关键上下文？
- [ ] 输出格式要求是否明确？
- [ ] 是否使用了 AI 不熟悉的术语？
- [ ] 任务是否过于复杂？（尝试分解）
- [ ] 是否提供了足够的背景信息？
