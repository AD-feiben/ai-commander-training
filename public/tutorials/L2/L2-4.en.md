# Let AI Help You Review Specifications



## Why AI Review is Needed?

Specifications written by humans always have omissions. AI review can help you:

- **Discover blind spots**: Find edge cases you didn't think of
- **Check completeness**: Ensure no key information is missed
- **Improve quality**: Make specifications clearer and more executable
- **Save time**: Automate the review process

## Dimensions of AI Review

### 1. Clarity Check

AI checks if specifications are clear and unambiguous:

```markdown
вқҢ Vague description:
"Users can log in quickly"

AI points out:
- "Quickly" is how fast? Need specific values
- Suggested change: "Login response time < 200ms"
```

### 2. Completeness Check

AI checks if important information is missing:

```markdown
вқҢ Incomplete specification:
## Input
- username
- password

AI points out:
- Missing type definitions (string? number?)
- Missing length limits
- Missing format validation
- Missing required/optional markers
```

### 3. Edge Case Check

AI helps you think about edge cases:

```markdown
вқҢ Edge cases not considered:
"Implement array sorting function"

AI points out edge cases:
- How to handle empty arrays?
- What if array has only one element?
- What if array elements are of inconsistent types?
- What if array contains null/undefined?
- What if array length exceeds limit?
```

### 4. Verifiability Check

AI checks if specifications are verifiable:

```markdown
вқҢ Not verifiable:
"Interface should be aesthetically pleasing"

AI suggests changing to:
- Use design system components
- Comply with WCAG 2.1 AA standards
- Pass design review
```

## AI Review Prompt Templates

### Basic Review Template

```markdown
Please review the following technical specification and point out possible issues:

## Specification Content
[Paste your specification]

## Please Check
1. Whether it is clear and unambiguous
2. Whether edge cases are complete
3. Whether error handling is sufficient
4. Whether there are missing scenarios
5. Whether specifications are verifiable
6. Whether there are technical feasibility issues

## Output Format
- Issue list (each including location, issue, suggestion)
- Improved specification version
```

### Deep Review Template

```markdown
As a senior architect, please deeply review the following technical specification:

## Specification Content
[Paste your specification]

## Review Dimensions

### 1. Functionality
- Whether functional description is complete
- Whether business logic is clear
- Whether user scenarios are thoroughly considered

### 2. Technical Feasibility
- Whether technical solution is reasonable
- Whether performance requirements are achievable
- Whether dependencies are clear

### 3. Security
- Whether there are security risks
- Whether permission control is in place
- Whether data protection is sufficient

### 4. Maintainability
- Whether code structure is clear
- Whether it is easy to extend
- Whether documentation is complete

### 5. Test Coverage
- Whether it is testable
- Whether test scenarios are complete
- Whether there are acceptance criteria

## Output Requirements
1. List discovered issues by dimension
2. Each issue includes: severity (High/Medium/Low), specific location, improvement suggestion
3. Provide optimized complete specification
4. If there are major design issues, provide alternative solutions
```

## Practical Examples

### Example 1: User Registration Specification Review

**Original Specification:**
```markdown
# User Registration Feature

## Input
- username
- password
- email

## Function
User registers an account
```

**AI Review Result:**

```markdown
## Issues Found in Review

### High Priority

1. **Missing Type Definitions**
   - Location: Input section
   - Issue: username, password, email all don't have type definitions
   - Suggestion: Define the type for each field

2. **Missing Security Requirements**
   - Location: Functional description
   - Issue: No password strength requirements
   - Suggestion: Add password complexity rules

3. **Missing Error Handling**
   - Location: Global
   - Issue: No description of how various error situations are handled
   - Suggestion: Add error response definitions

### Medium Priority

4. **Missing Length Limits**
   - username and password have no length limits
   - Suggestion: username 4-20 characters, password 8-32 characters

5. **Missing Format Validation**
   - email has no format requirements
   - Suggestion: Use standard email format validation

6. **Missing Uniqueness Constraints**
   - No description of whether username/email is unique
   - Suggestion: Clearly state uniqueness requirements

### Low Priority

7. **Missing Log Recording Requirements**
   - Suggestion: Record registration logs for audit purposes

## Improved Specification

#### User Registration Feature

#### Input

| Field | Type | Required | Validation Rules |
|------|------|------|----------|
| username | string | Yes | 4-20 characters, letters numbers underscores, unique |
| password | string | Yes | 8-32 characters, containing uppercase and lowercase and numbers |
| email | string | Yes | Valid email format, unique |

#### Functional Description

1. Validate input parameter formats
2. Check if username/email already exists
3. Encrypt password with bcrypt (cost=12)
4. Create user record
5. Send verification email (async)
6. Return user information (excluding password)

#### Error Handling

- 400: Parameter validation failed
- 409: Username or email already exists
- 500: Internal server error

#### Security Requirements

- Password encrypted storage
- Prevent brute force (rate limiting)
- Sensitive operations log records

```

### Example 2: API Interface Specification Review

**Original Specification:**
```markdown
## Get User List

GET /api/users

Returns user list
```

**AI Review Result:**

```markdown
## Issues Found in Review

### High Priority

1. **Missing Pagination Parameters**
   - Issue: No pagination, will cause problems with large data
   - Suggestion: Add page and size parameters

2. **Missing Authentication Requirements**
   - Issue: No description of whether login is required
   - Suggestion: Clearly state authentication requirements

3. **Missing Response Format**
   - Issue: No definition of returned data structure
   - Suggestion: Provide complete response example

### Medium Priority

4. **Missing Filter Conditions**
   - Issue: Cannot filter users by conditions
   - Suggestion: Add optional filter parameters

5. **Missing Sorting Description**
   - Issue: Return order is uncertain
   - Suggestion: Default sorting rules, support custom sorting

## Improved Specification

#### Get User List

GET /api/users?page=1&size=20&sort=created_at&order=desc

#### Authentication
Bearer Token required

#### Parameters

| Parameter | Type | Required | Description |
|------|------|------|------|
| page | number | No | Page number, default 1 |
| size | number | No | Items per page, default 20, max 100 |
| sort | string | No | Sort field, default created_at |
| order | string | No | Sort direction, asc/desc, default desc |
| status | string | No | Filter by status |

#### Response

```json
{
  "data": [
    {
      "id": "uuid",
      "username": "string",
      "email": "string",
      "status": "active",
      "created_at": "2025-01-15T10:30:00Z"
    }
  ],
  "pagination": {
    "page": 1,
    "size": 20,
    "total": 100,
    "total_pages": 5
  }
}
```

### Error Codes

- 401: Unauthorized
- 403: Forbidden
- 500: Server error

## Review Checklist

Before having AI review, self-check first:

- [ ] Whether specification includes all necessary chapters
- [ ] Whether input/output are clearly defined
- [ ] Whether edge cases are thoroughly considered
- [ ] Whether error handling is complete
- [ ] Whether there are specific acceptance criteria
- [ ] Whether technical solution is feasible

## Best Practices

### 1. Multiple Iterations

Don't expect one review to be perfect; typically 2-3 rounds of iteration are needed.

### 2. Review from Different Angles

- Round 1: Functionality review
- Round 2: Technical review
- Round 3: Security review

### 3. Record Common Issues

Build your team's specification review issue library to avoid repeating mistakes.

### 4. Combine with Manual Review

After AI review, manual final confirmation is still needed, especially for business logic parts.

## Tool Recommendations

- **Claude Code**: Deep review, detailed suggestions
- **ChatGPT**: Quick review, check basic issues
- **Cursor**: Real-time review within IDE
