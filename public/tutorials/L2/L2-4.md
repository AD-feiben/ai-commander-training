# 让 AI 帮你审查规范



## 为什么需要 AI 审查？

人写的规范总会有疏漏。AI 审查可以帮助你：

- **发现盲点**：找出你没想到的边界条件
- **检查完整性**：确保没有遗漏关键信息
- **提升质量**：让规范更清晰、更可执行
- **节省时间**：自动化审查流程

## AI 审查的维度

### 1. 清晰度检查

AI 会检查规范是否清晰无歧义：

```markdown
❌ 模糊描述：
"用户可以快速登录"

AI 指出：
- "快速"是多快？需要具体数值
- 建议改为："登录响应时间 < 200ms"
```

### 2. 完整性检查

AI 会检查是否遗漏重要信息：

```markdown
❌ 不完整规范：
## 输入
- username
- password

AI 指出：
- 缺少类型定义（string? number?）
- 缺少长度限制
- 缺少格式验证
- 缺少必填/可选标记
```

### 3. 边界条件检查

AI 会帮你思考边界情况：

```markdown
❌ 未考虑边界：
"实现数组排序功能"

AI 指出边界情况：
- 空数组怎么处理？
- 数组只有一个元素？
- 数组元素类型不一致？
- 数组中有 null/undefined？
- 数组长度超过限制？
```

### 4. 可验证性检查

AI 会检查规范是否可以验证：

```markdown
❌ 不可验证：
"界面要美观"

AI 建议改为：
- 使用设计系统组件
- 符合 WCAG 2.1 AA 标准
- 通过设计评审
```

## AI 审查提示词模板

### 基础审查模板

```markdown
请审查以下技术规范，指出可能存在的问题：

## 规范内容
[粘贴你的规范]

## 请检查
1. 是否清晰无歧义
2. 边界条件是否完整
3. 错误处理是否充分
4. 是否有遗漏的场景
5. 规范是否可验证
6. 是否有技术可行性问题

## 输出格式
- 问题列表（每条包含位置、问题、建议）
- 改进后的规范版本
```

### 深度审查模板

```markdown
作为资深架构师，请深度审查以下技术规范：

## 规范内容
[粘贴你的规范]

## 审查维度

### 1. 功能性
- 功能描述是否完整
- 业务逻辑是否清晰
- 用户场景是否考虑周全

### 2. 技术可行性
- 技术方案是否合理
- 性能要求是否可达
- 依赖关系是否明确

### 3. 安全性
- 是否有安全风险
- 权限控制是否到位
- 数据保护是否充分

### 4. 可维护性
- 代码结构是否清晰
- 是否易于扩展
- 文档是否完整

### 5. 测试覆盖
- 是否可测试
- 测试场景是否完整
- 是否有验收标准

## 输出要求
1. 按维度列出发现的问题
2. 每个问题给出：严重程度（高/中/低）、具体位置、改进建议
3. 提供优化后的完整规范
4. 如有重大设计问题，提供替代方案
```

## 实战示例

### 示例 1：用户注册规范审查

**原始规范：**
```markdown
# 用户注册功能

## 输入
- username
- password
- email

## 功能
用户注册账号
```

**AI 审查结果：**

```markdown
## 审查发现的问题

### 高优先级

1. **缺少类型定义**
   - 位置：输入部分
   - 问题：username, password, email 都没有定义类型
   - 建议：明确每个字段的类型

2. **缺少安全要求**
   - 位置：功能描述
   - 问题：没有密码强度要求
   - 建议：添加密码复杂度规则

3. **缺少错误处理**
   - 位置：全局
   - 问题：没有说明各种错误情况的处理
   - 建议：添加错误响应定义

### 中优先级

4. **缺少长度限制**
   - username 和 password 没有长度限制
   - 建议：username 4-20 字符，password 8-32 字符

5. **缺少格式验证**
   - email 没有格式要求
   - 建议：使用标准邮箱格式验证

6. **缺少唯一性约束**
   - 没有说明 username/email 是否唯一
   - 建议：明确唯一性要求

### 低优先级

7. **缺少日志记录要求**
   - 建议：记录注册日志，便于审计

## 改进后的规范

```markdown
# 用户注册功能

## 输入

| 字段 | 类型 | 必填 | 验证规则 |
|------|------|------|----------|
| username | string | 是 | 4-20字符，字母数字下划线，唯一 |
| password | string | 是 | 8-32字符，含大小写和数字 |
| email | string | 是 | 有效邮箱格式，唯一 |

## 功能描述

1. 验证输入参数格式
2. 检查 username/email 是否已存在
3. 使用 bcrypt 加密密码（cost=12）
4. 创建用户记录
5. 发送验证邮件（异步）
6. 返回用户信息（不含密码）

## 错误处理

- 400：参数验证失败
- 409：username 或 email 已存在
- 500：服务器内部错误

## 安全要求

- 密码加密存储
- 防止暴力破解（限流）
- 敏感操作记录日志
```
```

### 示例 2：API 接口规范审查

**原始规范：**
```markdown
## 获取用户列表

GET /api/users

返回用户列表
```

**AI 审查结果：**

```markdown
## 审查发现的问题

### 高优先级

1. **缺少分页参数**
   - 问题：没有分页，数据量大时会出问题
   - 建议：添加 page 和 size 参数

2. **缺少认证要求**
   - 问题：没有说明是否需要登录
   - 建议：明确认证要求

3. **缺少响应格式**
   - 问题：没有定义返回数据结构
   - 建议：提供完整的响应示例

### 中优先级

4. **缺少过滤条件**
   - 问题：不能按条件筛选用户
   - 建议：添加可选的过滤参数

5. **缺少排序说明**
   - 问题：返回顺序不确定
   - 建议：默认排序规则，支持自定义排序

## 改进后的规范

```markdown
## 获取用户列表

GET /api/users?page=1&size=20&sort=created_at&order=desc

### 认证
需要 Bearer Token

### 参数

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| page | number | 否 | 页码，默认 1 |
| size | number | 否 | 每页数量，默认 20，最大 100 |
| sort | string | 否 | 排序字段，默认 created_at |
| order | string | 否 | 排序方向，asc/desc，默认 desc |
| status | string | 否 | 过滤状态 |

### 响应

```json
{
  "data": [
    {
      "id": "uuid",
      "username": "string",
      "email": "string",
      "status": "active",
      "created_at": "2025-01-15T10:30:00Z"
    }
  ],
  "pagination": {
    "page": 1,
    "size": 20,
    "total": 100,
    "total_pages": 5
  }
}
```

### 错误码

- 401：未授权
- 403：无权限
- 500：服务器错误
```
```

```
┌─────────────┐
│  编写规范    │
└──────┬──────┘
       ↓
┌─────────────┐
│  AI 初审    │ ← 使用基础审查模板
└──────┬──────┘
       ↓
┌─────────────┐
│  修改完善    │
└──────┬──────┘
       ↓
┌─────────────┐
│  AI 深度审查 │ ← 使用深度审查模板
└──────┬──────┘
       ↓
┌─────────────┐
│  最终修改    │
└──────┬──────┘
       ↓
┌─────────────┐
│  规范定稿    │
└─────────────┘
```

## 审查检查清单

在让 AI 审查前，先自检：

- [ ] 规范是否包含所有必要章节
- [ ] 输入输出是否定义清晰
- [ ] 边界条件是否考虑周全
- [ ] 错误处理是否完整
- [ ] 是否有具体的验收标准
- [ ] 技术方案是否可行

## 最佳实践

### 1. 多次迭代

不要期望一次审查就完美，通常需要 2-3 轮迭代。

### 2. 不同角度审查

- 第一轮：功能性审查
- 第二轮：技术性审查
- 第三轮：安全性审查

### 3. 记录常见问题

建立团队的规范审查问题库，避免重复犯错。

### 4. 结合人工审查

AI 审查后，还需要人工最终确认，特别是业务逻辑部分。
