# 实战：写一个完整的功能规范



## 练习目标

本节将带你完整地编写一个功能规范，使用我们学到的所有知识。通过实战，你将掌握规范的完整编写流程。

## 案例：文件上传功能

### 原始需求

```
做一个文件上传功能：
- 支持拖拽上传
- 限制文件大小 10MB
- 支持图片和文档
- 显示上传进度
- 上传成功后显示预览
```

## 第一步：需求分析

### 提取功能点

| 需求 | 功能点 | 优先级 |
|------|--------|--------|
| 拖拽上传 | 拖拽交互、文件选择 | P0 |
| 大小限制 10MB | 文件验证 | P0 |
| 图片和文档 | 文件类型过滤 | P0 |
| 显示进度 | 进度反馈 | P0 |
| 显示预览 | 预览功能 | P1 |

### 识别隐含需求

- 上传前需要验证文件
- 需要错误处理（网络、服务器错误）
- 支持取消上传
- 多文件上传
- 上传历史记录

## 第二步：编写技术规范

```markdown
# 文件上传功能规范

---
版本: 1.0.0
日期: 2025-01-15
状态: 评审中
---

## 1. 功能概述

### 背景
用户需要上传图片和文档到系统，用于内容管理和分享。

### 目标
提供简单、直观的文件上传体验，支持拖拽和点击选择。

## 2. 功能需求

### 2.1 核心功能

#### 文件选择
- 支持点击选择文件
- 支持拖拽文件到上传区域
- 支持多文件同时上传（最多 5 个）

#### 文件验证
- 文件大小：≤ 10MB
- 文件类型：
  - 图片：image/jpeg, image/png, image/gif, image/webp
  - 文档：application/pdf, .doc, .docx
- 文件名长度：≤ 100 字符

#### 上传过程
- 显示上传进度条（百分比）
- 显示上传速度
- 支持取消上传
- 最大并发上传：3 个

#### 上传结果
- 成功：显示文件预览（图片）或图标（文档）
- 失败：显示错误信息，支持重试
- 提供文件访问链接

### 2.2 用户界面

#### 上传区域
- 默认状态：显示拖拽提示和选择按钮
- 拖拽中：高亮显示
- 验证中：显示验证进度
- 上传中：显示进度条
- 完成：显示预览和成功提示

#### 文件列表
- 显示文件名、大小、状态
- 支持删除已上传文件
- 支持重新上传失败文件

## 3. 技术规范

### 3.1 组件设计

```typescript
// UploadComponent Props
interface UploadProps {
  accept?: string[];           // 允许的文件类型
  maxSize?: number;            // 最大文件大小（MB）
  maxCount?: number;           // 最大文件数量
  multiple?: boolean;          // 是否支持多文件
  onUpload?: (files: File[]) => Promise<UploadResult[]>;
  onError?: (error: UploadError) => void;
  onProgress?: (progress: number) => void;
}

// 上传结果
interface UploadResult {
  id: string;
  fileName: string;
  fileSize: number;
  fileType: string;
  url: string;
  previewUrl?: string;         // 图片预览 URL
  status: 'success' | 'error';
  errorMessage?: string;
}
```

### 3.2 API 接口

#### 上传文件
```
POST /api/upload
Content-Type: multipart/form-data

Request:
- file: File (form-data)

Response (200):
{
  "success": true,
  "data": {
    "id": "uuid",
    "fileName": "example.jpg",
    "fileSize": 1024000,
    "fileType": "image/jpeg",
    "url": "https://cdn.example.com/files/uuid.jpg",
    "previewUrl": "https://cdn.example.com/preview/uuid.jpg"
  }
}

Response (400):
{
  "success": false,
  "error": {
    "code": "INVALID_FILE",
    "message": "文件验证失败",
    "details": {
      "reason": "FILE_TOO_LARGE",
      "maxSize": "10MB",
      "actualSize": "15MB"
    }
  }
}
```

### 3.3 状态管理

```typescript
interface UploadState {
  files: UploadFile[];
  uploading: boolean;
  progress: number;
  error: UploadError | null;
}

interface UploadFile {
  id: string;
  file: File;
  status: 'pending' | 'uploading' | 'success' | 'error';
  progress: number;
  result?: UploadResult;
  error?: UploadError;
}
```

## 4. 边界条件

### 4.1 文件验证
- 空文件：提示"文件不能为空"
- 文件过大：提示"文件大小超过 10MB 限制"
- 文件类型不支持：提示"不支持此文件类型"
- 文件名过长：自动截断或提示重命名

### 4.2 上传过程
- 网络中断：自动重试 3 次，失败后提示重试
- 服务器错误：提示"上传失败，请稍后重试"
- 用户取消：立即停止上传，清理资源
- 页面关闭：提示"上传未完成，确定要离开吗？"

### 4.3 并发控制
- 超过最大并发数：排队等待
- 用户上传更多文件：添加到队列

## 5. 性能要求

- 上传速度：充分利用带宽
- 内存占用：大文件分片上传，避免内存溢出
- 响应时间：上传接口响应 < 200ms

## 6. 安全要求

- 文件类型白名单：只允许指定类型
- 文件大小限制：防止超大文件攻击
- 文件名清理：防止路径遍历攻击
- 病毒扫描：服务器端扫描（可选）

## 7. 错误处理

| 错误码 | 场景 | 用户提示 |
|--------|------|----------|
| 400 | 文件验证失败 | 显示具体错误原因 |
| 413 | 文件过大 | "文件大小超过 10MB 限制" |
| 415 | 不支持的类型 | "不支持此文件类型" |
| 429 | 请求过于频繁 | "请稍后再试" |
| 500 | 服务器错误 | "上传失败，请稍后重试" |
| 网络错误 | 网络中断 | "网络错误，请检查网络连接" |

## 8. 验收标准

- [ ] 支持拖拽和点击选择文件
- [ ] 正确验证文件类型和大小
- [ ] 显示上传进度条
- [ ] 支持取消上传
- [ ] 图片显示预览，文档显示图标
- [ ] 正确处理所有错误情况
- [ ] 界面美观，响应式设计
- [ ] 代码通过代码审查
- [ ] 单元测试覆盖率 > 80%
```

## 第三步：AI 审查

将规范发给 AI 进行审查：

```markdown
请审查以下文件上传功能规范，从以下维度检查：

1. 功能完整性
2. 边界条件
3. 技术可行性
4. 安全性
5. 用户体验

[粘贴上面的规范]

请输出：
1. 发现的问题（按严重程度分类）
2. 改进建议
3. 优化后的规范
```

### AI 可能的反馈

```markdown
## 审查结果

### 高优先级

1. **缺少断点续传**
   - 大文件上传时，网络中断需要重新上传
   - 建议：添加分片上传和断点续传功能

2. **缺少上传历史**
   - 用户可能需要查看之前上传的文件
   - 建议：添加上传历史记录功能

### 中优先级

3. **图片压缩**
   - 大图片上传慢，占用存储空间大
   - 建议：客户端压缩图片后再上传

4. **重复文件检测**
   - 用户可能重复上传相同文件
   - 建议：基于文件哈希检测重复

### 已考虑周全

✅ 文件验证（类型、大小）
✅ 错误处理
✅ 并发控制
✅ 安全要求
```

## 第四步：迭代优化

根据 AI 的反馈，优化规范：

```markdown
# 优化后的规范（节选）

## 新增：分片上传

### 大文件处理
- 文件 > 5MB 时启用分片上传
- 分片大小：2MB
- 支持断点续传
- 自动重试失败的分片

### API 接口

#### 初始化上传
```
POST /api/upload/init
{
  "fileName": "example.jpg",
  "fileSize": 10485760,
  "fileType": "image/jpeg",
  "chunkSize": 2097152
}

Response:
{
  "uploadId": "uuid",
  "chunks": 5
}
```

#### 上传分片
```
POST /api/upload/chunk
Content-Type: multipart/form-data

- uploadId: uuid
- chunkIndex: 0
- chunk: Blob
```

#### 完成上传
```
POST /api/upload/complete
{
  "uploadId": "uuid"
}
```

## 新增：图片压缩

### 客户端压缩
- 图片 > 1MB 时自动压缩
- 压缩后质量：80%
- 最大尺寸：1920x1080

## 新增：重复检测

### 基于哈希的检测
- 计算文件 MD5
- 上传前检查是否已存在
- 存在则直接返回已有文件链接
```

## 实战总结

### 规范编写流程

```
1. 需求分析 → 提取功能点
2. 编写初稿 → 覆盖所有需求
3. AI 审查 → 发现问题
4. 迭代优化 → 完善规范
5. 规范定稿 → 团队确认
6. AI 实现 → 按规范开发
```

### 关键要点

1. **从需求出发**：不要遗漏隐含需求
2. **详细具体**：每个细节都要明确
3. **边界完整**：考虑所有异常情况
4. **可验证**：有明确的验收标准
5. **迭代优化**：通过审查不断完善
