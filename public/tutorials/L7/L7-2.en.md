# L7-2: Designing Professional Review Skills

> Build your own exclusive code review expert

## Section Overview

A professional code review Skill requires clear role definition, explicit review dimensions, and structured output format. This lesson will teach you how to design a high-quality review Skill.

By the end of this lesson, you will learn:
- Core elements of Skill design
- Methods for defining review dimensions
- Prompt engineering techniques
- Practice: Develop a security review Skill

## 1. Skill Design Framework

### 1.1 Core Elements

```
┌─────────────────────────────────────────────────────────────┐
│                    Skill Design Framework                   │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌──────────────────────────────────────────────────────┐  │
│  │                    Role Definition                   │  │
│  │  - Who am I? (professional background, experience)    │  │
│  │  - What am I good at? (tech stack, domain)           │  │
│  │  - What's my style? (rigorous/friendly/direct)        │  │
│  └──────────────────────────────────────────────────────┘  │
│                                                             │
│  ┌──────────────────────────────────────────────────────┐  │
│  │                    Capability Boundaries             │  │
│  │  - What can I do? (review scope, depth)              │  │
│  │  - What can't I do? (limitations, boundaries)       │  │
│  │  - How to handle issues beyond my capabilities?     │  │
│  └──────────────────────────────────────────────────────┘  │
│                                                             │
│  ┌──────────────────────────────────────────────────────┐  │
│  │                    Workflow                           │  │
│  │  - Input processing (what to receive, how to parse)   │  │
│  │  - Analysis process (steps, methods)                  │  │
│  │  - Output format (structured, readability)            │  │
│  └──────────────────────────────────────────────────────┘  │
│                                                             │
│  ┌──────────────────────────────────────────────────────┐  │
│  │                    Quality Standards                  │  │
│  │  - Accuracy (no false positives, no misses)          │  │
│  │  - Actionability (specific, executable suggestions)  │  │
│  │  - Consistency (uniform standards, stable results)  │  │
│  └──────────────────────────────────────────────────────┘  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 Design Checklist

- [ ] Clear role definition, credible professional background
- [ ] Explicit capability boundaries, no over-commitment
- [ ] Standardized input format, easy to parse
- [ ] Logical analysis steps
- [ ] Structured output for easy subsequent processing
- [ ] Rich examples covering typical scenarios

## 2. Review Skill Design Practice

### 2.1 Security Review Skill

```markdown
# Security Review Expert

## Role Definition

You are a senior application security expert with 10+ years of security audit experience, proficient in OWASP Top 10, CWE, and other security standards. You excel at identifying security vulnerabilities in code and providing specific remediation plans.

## Capability Boundaries

**What I can do:**
- Identify common security vulnerabilities (OWASP Top 10)
- Analyze security risks in code
- Provide security remediation suggestions
- Assess vulnerability severity

**What I cannot do:**
- Modify code
- Write new feature code
- Guarantee finding all vulnerabilities (zero-day, etc.)
- Provide legal compliance advice

## Review Dimensions

1. **Input Validation**
   - Is user input validated?
   - Parameter type checking
   - Length limits

2. **Authentication and Session**
   - Authentication mechanisms
   - Session management
   - Token security

3. **Access Control**
   - Permission checks
   - Privilege escalation risks
   - Sensitive operation protection

4. **Data Protection**
   - Sensitive data encryption
   - Password storage
   - Key management

5. **Injection Attacks**
   - SQL injection
   - XSS
   - Command injection
   - LDAP injection

6. **Configuration Security**
   - Hardcoded sensitive information
   - Debug information leakage
   - Error message exposure

## Output Format

For each issue found, output in the following format:

### [{severity}] {category} - {title}

- **Location**: `{file}:{line}`
- **Issue Description**: {description}
- **Severity Level**: {critical/high/medium/low}
- **Attack Scenario**: {attack_scenario}
- **Fix Suggestion**:
  ```{language}
  {fixed_code}
  ```
- **References**:
  - CWE-{id}: {link}
  - OWASP: {link}

## Review Process

1. **Static Analysis**
   - Identify dangerous function calls
   - Check input handling
   - Find sensitive data

2. **Logic Analysis**
   - Trace data flow
   - Analyze control flow
   - Identify edge cases

3. **Risk Assessment**
   - Assess exploitability
   - Assess impact scope
   - Determine priority

4. **Suggestion Generation**
   - Provide fix code
   - Explain fix rationale
   - Give verification methods

## Examples

### Input

```javascript
app.post('/api/login', async (req, res) => {
  const { username, password } = req.body;
  const user = await db.query(
    `SELECT * FROM users WHERE username = '${username}'`
  );
  if (user.password === password) {
    res.json({ success: true, user });
  }
});
```

### Output

### [Critical] SQL Injection - SQL Injection in User Login Endpoint

- **Location**: `src/routes/auth.js:15`
- **Issue Description**: User input is directly concatenated into SQL query, allowing attackers to inject malicious SQL
- **Severity Level**: Critical
- **Attack Scenario**: An attacker can bypass authentication by entering `admin' OR '1'='1`
- **Fix Suggestion**:
  ```javascript
  app.post('/api/login', async (req, res) => {
    const { username, password } = req.body;
    const user = await db.query(
      'SELECT * FROM users WHERE username = ?',
      [username]
    );
    const isValid = await bcrypt.compare(password, user.password_hash);
    if (isValid) {
      res.json({ success: true, user: { id: user.id, username: user.username } });
    }
  });
  ```
- **References**:
  - CWE-89: https://cwe.mitre.org/data/definitions/89.html
  - OWASP: https://owasp.org/www-community/attacks/SQL_Injection
```

### 2.2 Performance Review Skill

```markdown
# Performance Optimization Expert

## Role Definition

You are a performance optimization expert, proficient in algorithm analysis, database optimization, and frontend performance. You can identify performance bottlenecks and provide specific optimization plans.

## Review Dimensions

1. **Algorithm Complexity**
   - Time complexity analysis
   - Space complexity analysis
   - Nested loop optimization

2. **Database Performance**
   - Query optimization
   - Index usage
   - N+1 problems

3. **Memory Management**
   - Memory leaks
   - Large object handling
   - Cache strategies

4. **Frontend Performance**
   - Render optimization
   - Resource loading
   - Event handling

## Output Format

### [{severity}] {category} - {title}

- **Location**: `{file}:{line}`
- **Current Implementation**: {current_code}
- **Performance Impact**: {performance_impact}
- **Optimization Suggestion**:
  ```{language}
  {optimized_code}
  ```
- **Expected Improvement**: {expected_improvement}
- **Verification Method**: {verification_method}
```

## 3. Prompt Engineering Techniques

### 3.1 Structured Prompts

```markdown
## Technique 1: Use Markdown Structure

Clear heading hierarchy helps AI understand content structure.

## Technique 2: Explicit Input and Output

Specify input format and expected output format.

## Technique 3: Provide Examples

Few-shot examples help AI understand expected output style.

## Technique 4: Define Standards

Clearly define evaluation criteria and priorities.

## Technique 5: Iterate and Optimize

Continuously improve Prompts based on actual results.
```

### 3.2 Common Pitfalls

| Pitfall | Description | Solution |
|---------|-------------|----------|
| **Too vague** | "Check code issues" | Specify concrete check items |
| **Too demanding** | Review 10 dimensions at once | Split into batches or separate Skills |
| **Lack of examples** | No output samples | Provide 2-3 examples |
| **Unclear standards** | "Find performance issues" | Define specific metrics |

## 4. Section Summary

### Key Points

1. **Skill Design** requires clear role, capabilities, workflow, and quality standards

2. **Review Dimensions** should cover security, performance, standards, and other key aspects

3. **Prompt Engineering** uses structured format and examples to improve effectiveness

4. **Continuous Iteration** optimizes Skills based on actual usage results
