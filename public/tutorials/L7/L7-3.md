# L7-3: 审查规则定制

> 打造符合团队规范的审查标准

## 本节导读

每个团队都有自己的代码规范和质量标准。如何让 AI 代码审查符合团队的特定要求？本节课将教你如何定制审查规则，集成团队规范。

通过本节课，你将学会：
- 规则定制的核心方法
- 规则优先级管理
- 团队规范集成
- 实战：配置团队审查规范

## 一、规则体系设计

### 1.1 规则层级

```
规则层级结构：

Level 1: 基础规则（必须遵守）
├── 安全规则（Critical）
├── 语法规则（Error）
└── 类型规则（Error）

Level 2: 团队规范（应该遵守）
├── 命名规范（Warning）
├── 代码风格（Warning）
└── 文档规范（Warning）

Level 3: 最佳实践（建议遵守）
├── 设计模式（Suggestion）
├── 性能优化（Suggestion）
└── 可读性（Suggestion）

Level 4: 个性化规则（可选）
├── 团队偏好（Info）
└── 项目特定（Info）
```

### 1.2 规则配置格式

```yaml
# review-rules.yaml
rules:
  # 安全规则（Critical）
  security:
    level: critical
    enabled: true
    rules:
      - id: sql-injection
        name: SQL 注入检测
        severity: critical
        enabled: true
        patterns:
          - 'query\s*=\s*[`"].*\$\{.*\}[`"]'
        message: "发现潜在的 SQL 注入风险，请使用参数化查询"
        fix_suggestion: "使用参数化查询或 ORM"
      
      - id: xss-risk
        name: XSS 风险检测
        severity: critical
        enabled: true
        patterns:
          - 'innerHTML\s*=.*userInput'
        message: "发现 XSS 风险，请使用 textContent 或进行转义"
      
      - id: hardcoded-secret
        name: 硬编码密钥检测
        severity: high
        enabled: true
        patterns:
          - '(api[_-]?key|password|secret)\s*=\s*["\'][^"\']+["\']'
        message: "发现硬编码的敏感信息，请使用环境变量"
  
  # 性能规则（Warning）
  performance:
    level: warning
    enabled: true
    rules:
      - id: n-plus-one
        name: N+1 查询检测
        severity: warning
        enabled: true
        patterns:
          - 'for.*await.*find'
        message: "可能存在 N+1 查询问题，建议使用 JOIN 或预加载"
      
      - id: unnecessary-re-render
        name: 不必要的重渲染
        severity: suggestion
        enabled: true
        patterns:
          - 'useEffect.*\[\]'
        message: "检查 useEffect 依赖是否完整"
  
  # 规范规则（Warning）
  style:
    level: warning
    enabled: true
    rules:
      - id: naming-convention
        name: 命名规范检查
        severity: warning
        enabled: true
        conventions:
          functions: camelCase
          classes: PascalCase
          constants: UPPER_SNAKE_CASE
        message: "命名不符合团队规范"
      
      - id: max-line-length
        name: 行长度限制
        severity: suggestion
        enabled: true
        max_length: 100
        message: "行长度超过 100 字符，建议换行"
  
  # 自定义规则
  custom:
    level: info
    enabled: true
    rules:
      - id: no-console-in-prod
        name: 生产环境禁用 console
        severity: warning
        enabled: true
        patterns:
          - 'console\.(log|warn|error)'
        exclude_paths:
          - '**/*.test.ts'
          - '**/debug/**'
        message: "生产环境代码应避免使用 console"
```

## 二、规则优先级管理

### 2.1 优先级策略

```typescript
interface RulePriority {
  // 规则 ID
  ruleId: string;
  
  // 优先级级别
  level: 'critical' | 'high' | 'medium' | 'low' | 'info';
  
  // 是否阻塞（阻止合并）
  blocking: boolean;
  
  // 自动修复
  autoFix: boolean;
  
  // 忽略条件
  ignorePatterns?: string[];
  
  // 自定义消息
  customMessage?: string;
}

// 优先级配置
const priorityConfig: RulePriority[] = [

    ruleId: 'sql-injection',

    blocking: true,

  },

    ruleId: 'xss-risk',
    level: 'critical',

    autoFix: false
  },
  {
    ruleId: 'n-plus-one',
    level: 'warning',
    blocking: false,
    autoFix: false
  },
  {
    ruleId: 'naming-convention',
    level: 'warning',
    blocking: false,
    autoFix: true
```

### 2.2 规则覆盖策略

```
规则覆盖场景：

场景 1：全局规则 + 项目特定规则
├── .ai-review/rules.yaml（全局）
└── project/.ai-review/rules.yaml（项目特定）
    项目规则覆盖全局规则

场景 2：继承 + 扩展
base-rules.yaml
  └── extends: base-rules
      rules:
        - disable: rule-a
        - add: rule-b

场景 3：环境差异
development: 启用所有规则
production: 仅启用 critical 和 high
```

## 三、团队规范集成

### 3.1 规范来源

```
团队规范来源：

1. 编码规范文档
   ├── Airbnb JavaScript Style Guide
   ├── Google TypeScript Style Guide
   └── 团队自定义规范

2. 架构决策记录 (ADR)
   ├── 技术选型决策
   ├── 设计模式规范
   └── 接口设计规范

3. 历史审查记录
   ├── 常见问题汇总
   ├── 最佳实践案例
   └── 反面教材

4. 工具配置
   ├── .eslintrc
   ├── .prettierrc
   └── tsconfig.json
```

### 3.2 规范同步机制

```typescript
class TeamStandardSync {
  // 从 ESLint 配置同步规则
  async syncFromESLint(configPath: string): Promise<Rule[]> {
    const eslintConfig = await this.loadConfig(configPath);
    
    return eslintConfig.rules.map(([ruleId, severity]) => ({
      id: ruleId,
      level: this.mapSeverity(severity),
      source: 'eslint',
      enabled: severity !== 'off'
    }));
  }
  
  // 从代码规范文档提取规则
  async syncFromStyleGuide(docPath: string): Promise<Rule[]> {
    const doc = await this.loadDocument(docPath);
    
    // 使用 AI 提取规则
    const prompt = `
从以下代码规范文档中提取可自动检查的规则：

${doc}

输出格式：
- 规则名称
- 规则描述
- 检查方法
- 严重程度
    `;
    
    const rules = await this.aiExtractRules(prompt);
    return rules;
  }
  
  // 从历史审查记录学习
  async learnFromHistory(reviews: CodeReview[]): Promise<Rule[]> {
    // 分析问题模式
    const patterns = this.analyzePatterns(reviews);
    
    // 生成新规则
    return patterns.map(pattern => ({
      id: `learned-${pattern.id}`,
      name: pattern.name,
      level: pattern.severity,
      pattern: pattern.regex,
      message: pattern.suggestion,
      source: 'learned'
    }));
  }
}
```

## 四、实战：配置团队审查规范

### 4.1 完整配置示例

```yaml
# .ai-review/config.yaml
# AI 代码审查配置文件

# 基础设置
settings:
  # 审查语言
  languages:
    - typescript
    - javascript
    - react
  
  # 排除路径
  exclude:
    - 'node_modules/**'
    - 'dist/**'
    - '**/*.test.ts'
    - '**/*.spec.ts'
  
  # 包含路径（优先于排除）
  include:
    - 'src/**'
  
  # 最大文件大小（KB）
  max_file_size: 500
  
  # 并行审查数
  max_concurrent: 3

# 规则配置
rules:
  # 继承预设规则集
  extends:
    - '@ai-review/security'
    - '@ai-review/performance'
    - '@ai-review/react'
  
  # 禁用某些规则
  disable:
    - 'prefer-default-export'
    - 'no-console'
  
  # 自定义规则
  custom:
    # 团队特定的命名规范
    - id: team-naming
      name: 团队命名规范
      severity: warning
      pattern: |
        函数使用 camelCase
        类使用 PascalCase
        常量使用 UPPER_SNAKE_CASE
        私有方法使用 _ 前缀
      message: "命名不符合团队规范"
    
    # 特定业务规则
    - id: no-direct-api-call
      name: 禁止直接调用 API
      severity: error
      pattern: 'fetch\(|axios\.'
      exclude_paths:
        - 'src/api/**'
        - 'src/services/**'
      message: "请使用封装的 API 服务，不要直接调用 fetch/axios"
    
    # 组件规范
    - id: component-structure
      name: 组件结构规范
      severity: warning
      check: |
        1. 组件文件是否包含 Props 类型定义
        2. 是否使用函数组件
        3. 是否有默认导出
      message: "组件结构不符合规范"

# 审查流程
workflow:
  # 提交前审查
  pre_commit:
    enabled: true
    rules:
      - level: critical
      - level: error
  
  # PR 审查
  pull_request:
    enabled: true
    rules:
      - level: critical
      - level: error
      - level: warning
    # 需要人工审查的文件
    require_human_review:
      - 'src/core/**'
      - 'src/security/**'
  
  # 定期审查
  scheduled:
    enabled: true
    cron: '0 2 * * *'  # 每天凌晨 2 点
    rules:
      - level: all
    # 生成报告
    report:
      format: markdown
      destination: 'reports/code-review-daily.md'

# 通知配置
notifications:
  # 严重问题通知
  critical:
    channels:
      - slack: '#dev-alerts'
      - email: 'tech-lead@company.com'
  
  # 日报
  daily_report:
    enabled: true
    time: '09:00'
    channel: slack: '#dev-reports'

# 集成配置
integrations:
  # GitHub

    enabled: true
    # 自动评论
```

### 4.2 规则验证

```typescript
class RuleValidator {
  // 验证规则配置
  validate(config: ReviewConfig): ValidationResult {
    const errors: string[] = [];
    const warnings: string[] = [];
    
    // 检查规则 ID 唯一性
    const ruleIds = new Set<string>();
    for (const rule of config.rules.custom) {
      if (ruleIds.has(rule.id)) {
        errors.push(`规则 ID 重复: ${rule.id}`);
      }
      ruleIds.add(rule.id);
    }
    
    // 检查规则完整性
    for (const rule of config.rules.custom) {
      if (!rule.name) {
        errors.push(`规则 ${rule.id} 缺少名称`);
      }
      if (!rule.severity) {
        errors.push(`规则 ${rule.id} 缺少严重级别`);
      }
      if (!rule.pattern && !rule.check) {
        warnings.push(`规则 ${rule.id} 缺少检查逻辑`);
      }
    }
    
    // 检查工作流配置
    if (config.workflow.pre_commit.enabled && 
        config.workflow.pre_commit.rules.length === 0) {
      warnings.push('pre_commit 启用但未配置规则');
    }
    
    return {
      valid: errors.length === 0,
      errors,
      warnings
    };
  }
  
  // 测试规则
  async testRule(rule: Rule, testCases: TestCase[]): Promise<TestResult> {
    const results: TestCaseResult[] = [];
    
    for (const testCase of testCases) {
      const violations = await this.applyRule(rule, testCase.code);
      const expected = testCase.shouldTrigger;
      const actual = violations.length > 0;
      
      results.push({
        name: testCase.name,
        passed: actual === expected,
        expected,
        actual,
        violations
      });
    }
    
    return {
      ruleId: rule.id,
      total: testCases.length,
      passed: results.filter(r => r.passed).length,
      failed: results.filter(r => !r.passed).length,
      details: results
    };
  }
}
```

## 五、本节小结

### 核心要点

1. **规则分层** 从基础规则到个性化规则，层层递进

2. **优先级管理** 区分阻塞性规则和非阻塞性规则

3. **规范集成** 从现有工具和文档同步规则

4. **持续优化** 根据历史数据学习和改进规则
