# Spec Decomposition: From Large Specs to Small Specs

## Large Spec → Small Spec

Facing complex features, we need to decompose large specs into small specs, just like decomposing large tasks into small tasks.

## Decomposition Levels

### Level 1: Project Spec

```markdown
# E-commerce Website Project Spec

## Project Goal
Build a complete e-commerce website

## Feature Modules
1. User module
2. Product module
3. Shopping cart module
4. Order module
5. Payment module
6. Admin panel

## Tech Stack
- Frontend: React + TypeScript
- Backend: Node.js
- Database: PostgreSQL
```

### Level 2: Module Spec

```markdown
# Product Module Spec

## Feature List
1. Product list (pagination, filtering)
2. Product details
3. Product search
4. Product categories
5. Product management (CRUD)

## Data Model
Product: id, name, description, price, category_id, stock, images
```

### Level 3: Feature Spec

```markdown
# Product List Feature Spec

## Function
Display product list, support pagination and filtering

## Input

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| page | number | No | 1 | Page number |
| pageSize | number | No | 20 | Items per page |
| categoryId | string | No | - | Category ID |
| keyword | string | No | - | Search keyword |
| sort | string | No | newest | Sort method |

## Output

| Field | Type | Description |
|-------|------|-------------|
| list | Product[] | Product list |
| total | number | Total count |
| page | number | Current page |
| pageSize | number | Items per page |
```

## Benefits of Decomposition

1. **Reduces complexity** - Handle one small feature at a time
2. **Clarifies dependencies** - Know what to do first, what to do later
3. **Facilitates testing** - Each small feature can be tested independently
4. **Enables team collaboration** - Different people can be responsible for different parts
