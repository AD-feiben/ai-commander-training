# 规范分解：从大规范到小规范



## 为什么需要规范分解？

面对复杂项目，一份大规范往往包含数百个功能点。直接让 AI 实现整个规范会导致：

- **上下文超载**：AI 无法同时处理太多信息
- **理解偏差**：遗漏关键细节
- **质量下降**：代码质量难以保证
- **难以调试**：出问题后难以定位

**解决方案：把大规范分解成小规范，逐个实现。**

## 规范分解的三层结构

```
项目规范（宏观）
    ↓ 分解为
模块规范（中观）
    ↓ 分解为
功能规范（微观）
```

### 第一层：项目规范

**定位**：项目的总体蓝图

**内容**：
- 项目目标和背景
- 功能模块划分
- 技术架构选型
- 开发计划和里程碑

**示例**：
```markdown
# 电商网站项目规范

## 项目目标
构建一个 B2C 电商网站，支持商品展示、购物车、订单管理。

## 功能模块
1. 用户模块（注册/登录/个人中心）
2. 商品模块（展示/搜索/分类）
3. 购物车模块
4. 订单模块
5. 支付模块
6. 管理后台

## 技术栈
- 前端：React + TypeScript + Tailwind CSS
- 后端：Node.js + Express
- 数据库：PostgreSQL
- 缓存：Redis

## 里程碑
- 第 1 周：用户模块
- 第 2 周：商品模块
- 第 3 周：购物车和订单
- 第 4 周：支付和后台
```

### 第二层：模块规范

**定位**：单个模块的详细设计

**内容**：
- 模块职责和边界
- 功能列表
- 接口定义
- 数据模型

**示例**：
```markdown
# 商品模块规范

## 模块职责
负责商品的展示、搜索、分类管理。

## 功能列表
1. 商品列表（分页、筛选、排序）
2. 商品详情
3. 商品搜索（关键词、分类）
4. 商品分类（多级分类）
5. 商品管理（CRUD）- 后台

## 数据模型

### Product
- id: UUID
- name: string (商品名称)
- description: text (商品描述)
- price: decimal (价格)
- original_price: decimal (原价)
- stock: int (库存)
- category_id: UUID (分类ID)
- images: string[] (图片URL列表)
- status: enum (上架/下架)
- created_at: timestamp
- updated_at: timestamp

### Category
- id: UUID
- name: string
- parent_id: UUID (父分类ID，支持多级)
- sort_order: int

## 接口列表
- GET /api/products - 商品列表
- GET /api/products/:id - 商品详情
- GET /api/products/search - 商品搜索
- GET /api/categories - 分类列表
```

### 第三层：功能规范

**定位**：单个功能的详细实现规范

**内容**：
- 功能描述
- 输入输出
- 业务逻辑
- 边界条件
- 验收标准

**示例**：
```markdown
# 商品列表功能规范

## 功能描述
展示商品列表，支持分页、分类筛选、关键词搜索、排序。

## 接口定义

### GET /api/products

#### 请求参数

| 参数 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| page | number | 否 | 1 | 页码 |
| pageSize | number | 否 | 20 | 每页数量，最大 100 |
| categoryId | string | 否 | - | 分类 ID |
| keyword | string | 否 | - | 搜索关键词 |
| sort | string | 否 | created_at | 排序字段 |
| order | string | 否 | desc | 排序方向：asc/desc |
| minPrice | number | 否 | - | 最低价格 |
| maxPrice | number | 否 | - | 最高价格 |

#### 响应数据

```json
{
  "success": true,
  "data": {
    "list": [
      {
        "id": "uuid",
        "name": "商品名称",
        "price": 99.99,
        "original_price": 129.99,
        "image": "https://cdn.example.com/image.jpg",
        "stock": 100,
        "category": {
          "id": "uuid",
          "name": "分类名称"
        }
      }
    ],
    "pagination": {
      "page": 1,
      "pageSize": 20,
      "total": 150,
      "totalPages": 8
    }
  }
}
```

## 业务逻辑

1. 参数验证
   - page 和 pageSize 必须是正整数
   - keyword 长度不超过 100
   - minPrice 和 maxPrice 必须是非负数

2. 查询逻辑
   - 基础查询：status = 'active'
   - 分类筛选：category_id = categoryId 或其子分类
   - 关键词搜索：name 或 description 包含 keyword
   - 价格筛选：price BETWEEN minPrice AND maxPrice

3. 排序逻辑
   - 支持字段：created_at, price, sales
   - 默认按 created_at desc

4. 分页逻辑
   - 计算 offset: (page - 1) * pageSize
   - 限制 pageSize 最大 100

## 边界条件

1. 无数据时返回空数组
2. page 超过总页数时返回最后一页
3. keyword 为空字符串时忽略搜索
4. 分类不存在时返回空数组

## 性能要求

- 响应时间 < 200ms
- 支持 1000 QPS
- 使用数据库索引优化查询

## 缓存策略

- 商品列表缓存 5 分钟
- 分类数据缓存 1 小时
- 更新商品时清除缓存
```

## 分解的好处

### 1. 降低复杂度

```
大规范：电商网站（50+ 功能点）
    ↓ 分解
模块规范：商品模块（5 个功能）
    ↓ 分解
功能规范：商品列表（1 个功能）

AI 可以专注于实现单个功能，质量更高
```

### 2. 明确依赖关系

```
项目依赖图：

用户模块
    ↓
商品模块
    ↓
购物车模块（依赖用户和商品）
    ↓
订单模块（依赖购物车）
    ↓
支付模块（依赖订单）
```

### 3. 便于并行开发

```
团队成员并行工作：

开发者 A：实现用户模块
开发者 B：实现商品模块
开发者 C：实现购物车模块

互不干扰，提高效率
```

### 4. 易于测试和调试

```
单个功能规范：
- 输入明确
- 输出明确
- 边界条件明确
- 可以独立测试

出问题容易定位：是商品列表的问题，不是整个电商系统的问题
```

## 分解的原则

### 原则 1：自上而下分解

```
先定义项目整体，再分解到模块，最后到功能。

不要从细节开始，容易迷失方向。
```

### 原则 2：保持完整性

```
分解后的子规范合起来应该等于原规范。

不要遗漏功能，也不要添加未定义的功能。
```

### 原则 3：控制粒度

```
功能规范的大小：
- 太小：管理成本高（如"创建按钮"）
- 太大：AI 难以处理（如"实现整个购物车"）
- 合适：1-2 天的工作量（如"商品列表接口"）
```

### 原则 4：明确接口

```
模块之间通过接口通信，接口定义要明确：

模块 A --调用--> 接口 X --返回--> 模块 B

接口 X 的规范：
- 输入参数
- 输出格式
- 错误处理
- 性能要求
```

## 分解实战：电商网站

### Step 1：项目规范

```markdown
# 电商网站项目规范

## 功能模块
1. 用户模块
2. 商品模块
3. 购物车模块
4. 订单模块
5. 支付模块
```

### Step 2：模块规范

为每个模块编写规范：

```markdown
# 用户模块规范
## 功能列表
- 用户注册
- 用户登录
- 个人中心
- 地址管理

# 商品模块规范
## 功能列表
- 商品列表
- 商品详情
- 商品搜索
- 商品分类
```

### Step 3：功能规范

为每个功能编写详细规范：

```markdown
# 用户注册功能规范
## 接口
POST /api/auth/register

## 输入
- email: string (必填)
- password: string (必填)
- confirmPassword: string (必填)

## 验证规则
- 邮箱格式正确
- 密码 8-32 位，含大小写和数字
- 两次密码一致
- 邮箱未被注册

## 输出
- 成功：返回用户信息和 token
- 失败：返回具体错误信息
```

### Step 4：逐个实现

```
Week 1:
- Day 1-2: 用户注册
- Day 3-4: 用户登录
- Day 5: 个人中心

Week 2:
- Day 1-2: 商品列表
- Day 3-4: 商品详情
- Day 5: 商品搜索

...
```

## 与 AI 协作的分解流程

### 第一步：让 AI 帮你分解

```markdown
我有一个电商网站的需求，请帮我分解规范：

## 需求
[项目需求描述]

## 请输出
1. 模块划分（MECE 原则）
2. 每个模块的功能列表
3. 模块间的依赖关系
4. 推荐的实现顺序
```

### 第二步：确认分解结果

检查 AI 的分解：
- 是否完整？
- 是否合理？
- 依赖关系是否正确？

### 第三步：逐个实现

```
你：请实现用户注册功能
AI：实现代码
你：审查代码
你：请实现用户登录功能
AI：实现代码
...
```

## 常见错误

### 错误 1：分解不彻底

```
❌ 模块规范太粗
商品模块：
- 商品相关功能

✅ 功能明确
商品模块：
- 商品列表
- 商品详情
- 商品搜索
- 商品分类
```

### 错误 2：粒度不一致

```
❌ 有的功能太细，有的太粗
- 创建按钮（太细）
- 实现购物车（太粗）

✅ 粒度一致
- 商品列表接口
- 添加购物车接口
- 提交订单接口
```

### 错误 3：忽视依赖

```
❌ 先做订单，后做用户
（订单依赖用户信息）

✅ 先做用户，后做订单
```
