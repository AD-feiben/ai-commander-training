# 用 AI 进行需求分析



## 为什么用 AI 做需求分析？

需求分析是软件开发的第一步，也是最关键的一步。传统方式下，需求分析需要：
- 经验丰富的产品经理
- 大量的沟通会议
- 反复的确认和修改

**AI 可以大幅提升需求分析的效率和质量。**

### AI 在需求分析中的优势

| 优势 | 说明 |
|------|------|
| **全面性** | AI 不会遗漏常见的功能点 |
| **结构化** | 自动按模块组织需求 |
| **技术视角** | 同时考虑技术实现可行性 |
| **快速迭代** | 几分钟完成一轮分析 |
| **成本低廉** | 不需要资深专家参与 |

## 需求分析的核心步骤

```
1. 收集原始需求 → 2. AI 分析扩展 → 3. 人工确认补充 → 4. 输出需求文档
```

## AI 需求分析提示词模板

### 基础版

```markdown
请帮我分析以下需求：

## 原始需求
[一句话描述，如：做一个在线商城]

## 请输出
1. 功能模块划分（MECE 原则）
2. 每个模块的核心功能
3. 数据模型设计
4. 技术选型建议
5. 潜在风险和注意事项
```

### 进阶版

```markdown
作为资深产品经理和架构师，请帮我深度分析以下需求：

## 项目背景
[业务背景、目标用户、核心价值]

## 核心需求
[主要功能描述]

## 约束条件
- 技术栈：[如 React + Node.js]
- 时间：[如 3 个月]
- 预算：[如 50 万]

## 请输出

### 1. 功能需求分析
- 核心功能（必须实现）
- 扩展功能（可以延后）
- 创新功能（差异化竞争）

### 2. 用户角色分析
- 用户画像
- 使用场景
- 痛点和需求

### 3. 系统架构建议
- 技术架构
- 数据架构
- 部署架构

### 4. 数据模型
- 核心实体
- 实体关系
- 关键字段

### 5. 非功能需求
- 性能要求
- 安全要求
- 可用性要求

### 6. 项目计划
- 里程碑划分
- 关键路径
- 风险点

### 7. 待确认问题
- 需要进一步明确的需求
- 需要业务方决策的问题
```

## 实战示例

### 示例 1：博客系统需求分析

**原始需求：**
```
做一个博客系统，用户可以写文章、评论。
```

**AI 分析结果：**

```markdown
## 1. 功能需求分析

### 核心功能（P0）
- 用户注册/登录
- 文章 CRUD
- 文章列表/详情
- 评论功能

### 扩展功能（P1）
- 文章分类/标签
- 文章搜索
- 用户关注
- 文章收藏

### 创新功能（P2）
- 协作编辑
- 文章版本历史
- AI 写作助手

## 2. 用户角色分析

### 读者
- 浏览文章
- 评论互动
- 关注作者

### 作者
- 发布文章
- 管理评论
- 查看统计

### 管理员
- 内容审核
- 用户管理
- 系统配置

## 3. 数据模型

### User
- id, username, email, password, avatar, bio, role

### Post
- id, title, content, author_id, status, views, created_at, updated_at

### Comment
- id, content, post_id, user_id, parent_id, created_at

### Category
- id, name, slug, description

## 4. 技术建议

- 前端：Next.js（SSR 利于 SEO）
- 后端：Node.js + Express
- 数据库：PostgreSQL
- 缓存：Redis
- 搜索：Elasticsearch
- 存储：AWS S3

## 5. 潜在风险

- SEO：需要服务端渲染
- 性能：大量文章需要分页和缓存
- 安全：XSS、SQL 注入防护
- 合规：用户数据隐私保护

## 6. 待确认问题

1. 是否支持 Markdown 编辑器？
2. 是否需要文章审核机制？
3. 评论是否需要审核？
4. 是否支持多语言？
5. 目标用户量是多少？
```

### 示例 2：电商系统需求分析

**原始需求：**
```
做一个电商网站，卖衣服。
```

**AI 分析结果（节选）：**

```markdown
## 核心功能模块

### 商品模块
- 商品发布（多规格、多图片）
- 商品分类（多级分类）
- 商品搜索（筛选、排序）
- 库存管理

### 订单模块
- 购物车
- 下单流程
- 订单管理
- 物流跟踪

### 支付模块
- 多种支付方式
- 支付安全
- 退款处理

### 营销模块
- 优惠券
- 满减活动
- 积分系统
- 会员等级

## 关键决策点

1. **库存扣减时机**
   - 方案 A：下单扣减（防止超卖）
   - 方案 B：支付扣减（提高转化率）

2. **价格计算**
   - 原价、促销价、会员价如何叠加？
   - 优惠券和满减能否同时使用？

3. **物流集成**
   - 自营物流还是第三方？
   - 运费如何计算？
```

## 人机协作流程

### 第一轮：AI 初步分析

```
你：提供原始需求
AI：输出初步分析
```

### 第二轮：人工补充

```
你：确认正确部分，补充遗漏信息
AI：基于补充信息重新分析
```

### 第三轮：深入讨论

```
你：针对关键问题深入提问
AI：提供详细方案和对比
```

### 第四轮：输出文档

```
AI：整理成标准需求文档
你：最终确认
```

## 需求分析检查清单

AI 分析后，检查是否包含：

- [ ] 所有功能模块
- [ ] 用户角色定义
- [ ] 核心业务流程
- [ ] 数据模型设计
- [ ] 非功能需求
- [ ] 潜在风险
- [ ] 待确认问题

## 常见错误

### 错误 1：需求过于模糊

```
❌ "做一个社交 APP"

结果：AI 无法给出有针对性的分析

✅ "做一个面向大学生的校园社交 APP，
     主要功能是二手交易和失物招领"
```

### 错误 2：忽视约束条件

```
❌ 不提供技术栈、时间、预算约束

结果：AI 给出的方案可能不切实际

✅ 明确约束："使用 React，3 个月，团队 5 人"
```

### 错误 3：不验证 AI 的输出

```
❌ 完全相信 AI 的分析

结果：可能包含不合理或不符合业务的建议

✅ 人工审核，结合业务知识判断
```

## 进阶技巧

### 技巧 1：对比分析

让 AI 提供多个方案对比：

```markdown
请提供 3 种技术架构方案，并对比：
- 方案 A：单体架构
- 方案 B：微服务架构
- 方案 C：Serverless 架构

对比维度：开发成本、维护成本、扩展性、性能
```

### 技巧 2：反向提问

让 AI 主动发现问题：

```markdown
基于以上需求，你认为：
1. 最容易出现问题的环节是什么？
2. 最容易被忽视的需求是什么？
3. 对技术架构影响最大的因素是什么？
```

### 技巧 3：场景推演

让 AI 模拟用户使用场景：

```markdown
请模拟以下用户场景，识别潜在问题：

场景 1：新用户首次使用
场景 2：高峰期大量并发
场景 3：网络不稳定情况
场景 4：恶意用户攻击
```

## 练习

### 练习 1：分析一个简单需求

**需求**：做一个待办事项 APP

**任务**：
1. 使用提示词模板让 AI 分析
2. 检查 AI 的输出是否完整
3. 补充你想到但 AI 遗漏的功能

### 练习 2：对比分析

**需求**：做一个在线文档协作工具

**任务**：
1. 让 AI 提供 2-3 种技术方案
2. 对比各方案的优缺点
3. 选择最适合的方案
